# 榜单浏览器采集与元数据校验

## 目标和边界

系统只采集公开榜单元数据，不采集小说正文。番茄使用可见浏览器渲染，出现私有字体字符时会先尝试本机真实 OCR；若未安装 OCR 引擎或 OCR 未能产出保守标题候选，则保存截图并进入 `ocr_required`。起点出现安全挑战时进入 `user_action_required`，由用户在真实浏览器完成。系统不破解验证码、不伪造签名、不轮换代理规避封禁，也不接入盗版书库。

合法交叉校验源首期为 Open Library。校验结果分为 `confirmed / partial_match / ambiguous / conflict / not_found / unavailable / unvalidated`；只能作为辅助证据，不能因为开放目录未收录中文网文就判定榜单数据错误或覆盖原榜单字段。

## 本地使用

首次安装浏览器运行时：

```bash
backend/.venv/bin/pip install -r backend/requirements.txt
backend/.venv/bin/playwright install chromium
```

生成采集工件：

```bash
backend/.venv/bin/python backend/scripts/capture_ranking.py qidian --output var/qidian.json
backend/.venv/bin/python backend/scripts/capture_ranking.py fanqie --output var/fanqie.json
backend/.venv/bin/python backend/scripts/capture_ranking.py zongheng --output var/zongheng.json --headless --no-pause
```

默认会打开可见浏览器并等待用户在真实浏览器完成登录或挑战后按 Enter。可调度的公开页面可以使用 `--headless --no-pause`，但遇到登录/验证码仍必须输出 `user_action_required`，不得绕过。番茄私有码字形会自动调用本机 `tesseract`（如已安装）并生成 `browser_ocr` 工件；未安装或 OCR 失败时返回 `ocr_required`。

然后向后端指定工件：

```bash
export RANKING_CAPTURE_QIDIAN_PATH="$PWD/var/qidian.json"
export RANKING_CAPTURE_FANQIE_PATH="$PWD/var/fanqie.json"
```

番茄工件若为 `ocr_required`，由独立 OCR worker 或人工授权工具读取截图，按同一 JSON 协议输出 `items`；OCR 每条必须包含 `confidence`，低于业务阈值的记录进入人工确认，不能自动进入市场分析。

也可以直接在前端「扫榜中心 → 导入已有榜单文件」上传浏览器/OCR 采集工件。系统会自动识别 `source/status/collector/items` 协议并调用：

- `POST /api/v1/ranking/capture-import?project_id=...`
- `POST /api/v1/ranking/snapshots/{snapshot_id}/confirm-capture`

这样番茄 OCR、起点用户会话、纵横或手动采集都走同一套入库、证据、置信度和人工确认门禁；不再需要把 `RANKING_CAPTURE_*_PATH` 写到服务端环境变量才能入库。环境变量路径仍保留给自动调度或服务器侧采集任务。

CSV 最低字段为 `rank,title`，可选 `author,category,url,readers,status,last_update`。每一行会记录文件绝对路径和行号，便于审计。

## 工件状态

| 状态 | 含义 | 后续动作 |
|---|---|---|
| `succeeded` | 已获得公开元数据 | 规范化、去重、保存快照 |
| `user_action_required` | 登录或安全挑战仍在页面 | 用户处理后重新采集 |
| `ocr_required` | 番茄标题含私有字体字符 | PaddleOCR/人工确认 |
| `schema_changed` | 页面结构不匹配 | 保存截图，修订选择器 |

任何非成功状态都必须形成失败快照，不允许返回空数组并宣称采集成功。

## 前端与接口

榜单中心支持上传 CSV/JSON，并调用 `POST /api/v1/ranking/import`。导入限制为 1–200 条，只有项目 owner/editor 可以写入。采集置信度低于 0.85 的快照标记为 `capture_status=needs_review`，在人工处理前禁止进入市场分析。

浏览器/OCR 工件导入调用 `POST /api/v1/ranking/capture-import`。非成功状态（`user_action_required`、`ocr_required`、`schema_changed`）会形成失败快照和证据，不会返回空成功；低置信 OCR 成功快照必须由 owner/editor 调用 `confirm-capture` 确认后才能进入市场分析。

`POST /api/v1/ranking/snapshots/{snapshot_id}/validate-metadata` 对快照条目执行 Open Library 交叉校验。详细证据合并写入 `ranking_items.metrics.validation`，不得覆盖原有 metrics；快照只保存汇总计数。`not_found` 与 `unavailable` 必须和真正的字段 `conflict` 分开显示。

数据库的 `ranking_snapshots.status` 只描述是否成功入库，`capture_status` 描述采集质量/人工接管状态，`ranking_items.metadata_status` 描述外部元数据校验结果，三者不得混用。
