## 1. 产品目标分析

### 1.1 最终定位（已确认，写入项目首页）

**NovelCraft Personal Studio — AI Content Studio for Writers & Creators**
一个人，也能拥有专业内容团队的生产力。

- 服务对象：个人创作者 + 1~5 人小团队；
- 完整闭环：选题 → 创作 → 审核 → 发布 → 数据反馈 → 持续优化；
- 明确不做：微服务、K8s、SaaS 多租户、企业组织架构、支付计费、复杂权限树。

**评审意见：定位成立且优于旧版两案。** 相比"个人版"，它给了协作留白；相比"AI 内容公司操作系统"，它砍掉了永远用不到的企业复杂度。1~5 人规模下，单 VPS + Compose 栈完全够用，架构决策全部可以按"垂直扩容优先"简化。

### 1.2 需求本质：250+ 功能 ≠ 250 个模块

最终清单共 13 大类约 250+ 项功能点。若逐项独立实现，单人开发是死路。**本报告最核心的分析结论：这 250+ 项功能可以完全收敛为 8 个平台能力 + 大量"配置态"功能**：

| # | 平台能力 | 承载的功能类别（示例） | 占清单比重 |
|---|---|---|---|
| C1 | **统一内容模型**（Everything is Content） | 长篇/短篇/微小说/公众号文/小红书笔记/知乎回答/短视频脚本/图文稿……全部是 `Content(type, body, meta)` 的不同 type | ~25% |
| C2 | **工作流引擎**（Automation First） | 自动连载、批量生产、定时生产、一稿多平台、暂停/恢复/取消/重试、人工确认节点、预设+自定义工作流 | ~15% |
| C3 | **AI Gateway + Agent 层** | 10 个 Agent、4 家模型路由/降级/重试、Token 预算、成本统计、模型对比、A/B、Prompt 实验室 | ~15% |
| C4 | **Knowledge Hub**（Everything is Knowledge） | 素材中心、RAG、向量检索、金句库/标题库/平台规则库/品牌风格库、项目级+全局知识库、风格学习、拆书分析 | ~12% |
| C5 | **通用版本系统**（Everything is Versioned） | 章节版本、内容版本树、人物/世界观/Prompt/知识版本、对比、恢复 | ~8% |
| C6 | **叙事一致性引擎**（小说独有护城河） | 伏笔、时间线、人物弧线、OOC、设定冲突、防崩、7 层上下文装配、分层摘要 | ~12% |
| C7 | **发布网关 + 数据回流** | 平台账号、定时/自动发布、格式适配、发布记录、阅读/点赞/收益回流、表现分析 | ~8% |
| C8 | **追踪与治理**（Everything is Traceable） | AI 调用全记录（输入/输出/Prompt/模型/Token/耗时/原因）、审计日志、操作日志 | ~5% |

其余为 UI 层功能（三栏工作台、选中改写、暗色模式等），落在设计系统与编辑器上。

**这是"功能一个不少 + 架构保持轻量"能同时成立的唯一解**：新增"小红书视频脚本"不是新模块，而是 C1 的一个 content type + C2 的一个工作流模板 + C3 的一个 prompt。平台能力做厚，具体功能做薄。

### 1.3 六大理念的工程含义（逐条确认可行性）

| 理念 | 评审结论 | 落地要点 |
|---|---|---|
| Everything is Content | ✅ 采纳，为全系统第一设计决策 | 单 `contents` 表 + type 判别 + JSONB meta；小说章节也是 content（详见架构评审 §2） |
| Everything is Knowledge | ✅ 采纳 | knowledge_items 统一入库（含人物/世界观/热点/文章/Prompt），pgvector 全局检索，AI 上下文装配从这里取料 |
| Everything is Traceable | ✅ 采纳，成本极低收益极高 | `ai_calls` 单表记录全部字段；这是 Prompt 实验室、模型对比、成本统计的同一数据源 |
| Everything is Versioned | ✅ 采纳 | 多态 `versions` 表（entity_type + entity_id + snapshot），一套机制覆盖正文/人物/世界观/Prompt/知识 |
| Automation First | ✅ 采纳 | 工作流引擎 + Celery beat 定时触发；"每天上午抓热点→生成三平台→等待审核"就是一条预设工作流 |
| Offline First | ✅ 采纳并定案（分三级，见 §4 R4 与架构评审 §9/§13.3） | Web 应用的"离线"= PWA + 本地缓存 + 任务出站队列；L1 只读 / L2 离线编辑（乐观锁 + 三方对比 UI）/ L3 AI 出站队列（幂等重放）；明确不做 CRDT/自动合并/实时协同，不承诺全功能离线 |

### 1.4 UI 第一（最高原则）的成本认知

对标 Cursor/Notion/Linear/Raycast 意味着：必须先建**设计系统**（tokens、组件库、暗色模式、动效规范），不能页面各写各的。这是一次性投入（约 1.5~2 周）+ 每页遵守成本。**结论：值得，且必须放在 MVP 之前做地基**，否则后续 30+ 页面返工成本远高于此。

## 2. 核心竞争力分析

排序不变但表述升级：

1. **叙事一致性引擎（C6）**——百万字不崩仍是唯一硬护城河，市面无可靠方案；
2. **闭环工作流（C2+C7）**——从灵感到数据回流的全自动流水线，竞品全部停留在"辅助写作"单点；
3. **一稿多平台（C1+C4）**——统一内容模型让"一部小说自动派生公众号/头条/知乎/小红书/视频脚本"成为一次 fan-out 操作，这是"一个人 = 一个团队"口号的直接兑现；
4. **可追踪可回滚（C5+C8）**——创作者的资产安全感，商业写作软件普遍缺失。

自媒体单篇生成、热点榜单本身仍是同质化能力，但在本定位下它们是**闭环的必要环节**而非卖点，按配置态低成本实现即可，不再建议砍除。

## 3. 当前设计优势

1. 定位收敛彻底：明确的"不做清单"砍掉了 80% 架构复杂度来源；
2. 六大理念相互咬合：Content 统一使复用可行，Knowledge 统一使 RAG 有料，Traceable 使 A/B 与成本优化有数据，Versioned 使自动化敢放手跑；
3. 技术栈零迁移：React + FastAPI + PostgreSQL/pgvector + Redis + Celery + Compose 与 V1 完全一致，106 个测试与 25 个修复全部保值；
4. 工作流引擎是"自定义功能"的出口：用户想要的任何新生产线，都可以由"自定义工作流 + 节点配置"自助完成，工程侧无需逐个开发；
5. 1~5 人协作用 Owner/Editor/Viewer 三角色即可覆盖，权限模型一天可设计完。

## 4. 当前设计风险

| # | 风险 | 等级 | 说明与对策 |
|---|---|---|---|
| R1 | **全量交付周期长** | 🔴 高 | 250+ 功能即使收敛为 8 能力，单人全量交付仍需 ~9 个月。对策：严格里程碑制（见路线图），每 4~10 周有一个日常可用版本，任何时点停下来系统都完整可用 |
| R2 | **自动发布 ToS/封号** | 🔴 高 | 起点/番茄/小红书/公众号自动化发布违反平台条款。**功能保留**，但默认模式为"人工确认节点在发布前"（工作流引擎天然支持），全自动为显式开启项并持续风险提示；Medium/Substack/X 等有官方 API 的平台优先做全自动 |
| R3 | **热点抓取数据源脆弱 + 合规** | 🟠 中高 | 榜单接口随时变。对策：抓取器做成可插拔 adapter + 失败降级为手动录入；单人自用规模合规风险可控，但不进任何分发产物 |
| R4 | **Offline First 范围蔓延** | 🟠 中 | "全功能离线"会拖入本地数据库同步冲突的深水区。对策：三级定义——L1 只读缓存（项目/章节/人物/知识/版本可浏览搜索）、L2 编辑离线暂存（IndexedDB + 恢复网络自动提交，冲突以服务端版本树解决）、L3 AI 任务出站队列。承诺 L1~L3，不做 CRDT 实时协同离线合并 |
| R5 | **UI 第一 vs 单人带宽** | 🟠 中 | 商业级 UI 是持续成本。对策：设计系统先行 + 页面按"工作台>编辑器>控制台>管理页"分级投入，管理页允许朴素 |
| R6 | **多 Agent 协作复杂化** | 🟡 中 | 10 个 Agent 若做成自治对话系统不可调试。对策：Agent = 工作流节点（角色化 Prompt + 工具 + Schema 输出），协作 = 工作流编排，保持确定性 |
| R7 | **成本失控** | 🟡 中 | 自动化程度越高烧钱越快。对策：C8 追踪 + 三级预算熔断（任务/项目/日）从 MVP 就内建 |

## 5. 缺失模块（清单之外必须补）

1. **设计系统**（UI 第一的地基，清单未列但必需）；
2. **内容安全前置过滤**（敏感词/违禁词，中国平台发布的硬前提）；
3. **备份恢复的 RPO/RTO 定义**（清单有"备份/恢复"字样，需定标：每日全量 + RPO≤24h、RTO≤1h，含异地副本）；
4. **告警通道**（任务失败/预算触顶/磁盘水位 → Telegram/邮件）；
5. **ROI 面板**（成本统计 + 数据回流已在清单，需明确二者合并为收益-成本对照视图，作为"持续优化"闭环的仪表盘）。

## 6. 技术难点（按投入排序）

1. 7 层上下文装配 + 分层摘要 + 实体状态硬约束（C6 核心，全项目最难）;
2. 工作流引擎的可靠执行语义：幂等、断点续跑、人工确认节点的挂起/唤醒、DAG 局部重试；
3. 通用版本树的多态设计与 diff/恢复性能（正文大文本 vs 结构化实体两套 diff 策略）；
4. 多模型 Prompt 等价性：同一 prompt 在 4 家模型上的输出结构差异，Prompt 库必须带 model 分支 + golden case 回归；
5. 风格学习/拆书分析：本质是"样本入库 → 特征提取 prompt → 风格卡（style card）→ 注入生成上下文"，需防止过拟合到侵权级模仿（输出相似度检测）；
6. 离线编辑的冲突处理（以版本树 + 三方合并提示解决，不做自动合并）。

## 7. 开发风险

1. 单人节奏下**里程碑门禁是唯一防烂尾机制**：M(n) 未验收，M(n+1) 不开工；
2. 平台能力（C1~C8）的接口一旦定错，返工是全局性的——因此 MVP 阶段就必须用最终形态的 Content/版本/追踪表结构，宁可 MVP 慢一周；
3. V1 演进策略不变：在现仓库上重构迁移，禁止另起炉灶；V1 的 novels/chapters 数据迁入统一 Content 模型是 MVP 的第一个迁移任务；
4. 文档纪律：本套 7 份文档即 V2 基线，原 37 份空壳文档随开发逐份补实，代码 PR 与文档 PR 同评审。

## 8. 结论

"功能一个不少，架构保持轻量"在工程上成立，充要条件是：**以 8 个平台能力为骨架、以配置态承载长尾功能、以里程碑门禁控制交付节奏**。方向确认通过，进入架构评审与路线执行。
