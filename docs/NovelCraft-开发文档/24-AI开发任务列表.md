# NovelCraft AI 开发任务列表

> 本文档是多 AI 按序开发与验收台账。功能优先于重构、包装和非阻塞优化。未取得可复现证据的功能一律不得上报完成。

## 1. 状态与验收等级

状态仅允许：`待核验`（可能已有代码但无完整证据）、`待开发`、`开发中`、`待验收`、`阻塞`、`已验收`、`已放弃`。创建本表时未逐项复验，故全部保守标为“待核验”。“已放弃”不等于完成，必须记录原因、影响和替代方案。

| 等级 | 最低验收要求 |
|---|---|
| T0 | 文件、入口、配置或结构静态存在；不能证明功能可用 |
| T1 | 单元测试、类型检查或静态检查通过 |
| T2 | 模块真实运行，关键输入输出与错误路径可复现 |
| T3 | 使用非伪造数据贯通前后端或真实上下游集成链路 |
| T4 | 从用户入口到结果落库/导出端到端跑通，含失败与恢复 |
| T5 | 生产或等价预发布验收，含监控、回滚、安全与数据对账 |

截图不能单独证明业务成功，Mock 结果不能作为 T3 及以上证据。依赖未验收时，不得声称下游任务完成。

## 3. 多 AI 协作规则

### 文件所有权

1. 任务只能由主 Agent 分配；每项同时只有一个负责人 AI。协作者须有明确子范围。
2. AI 只能修改任务列出的文件范围；跨界前必须停手，上报文件、原因和最小方案。
3. 两个 AI 禁止同时修改同一文件。公共文件由主 Agent 指定唯一所有者。
4. 开工前检查工作区改动，不得覆盖、回滚、格式化或顺手修改用户及其他 AI 的内容。

### 禁止虚假上报

1. 开发 AI 禁止自行标 `✅` 或“已验收”，禁止宣称“全部完成”“生产可用”。最多提交为“待验收”。
2. 未执行的命令不得写“通过”，未检查的页面不得写“正常”，静态代码和 Mock 不得冒充真实业务成功。
3. 失败、部分成功、环境缺失和权限不足必须原样上报；禁止裁掉失败输出、只截成功片段或用推测补结果。
4. 证据不可复现时，任务退回“开发中”或“待核验”。

### 主 Agent 验收

1. 只有主 Agent 可在独立复验后标“已验收”；不得以开发 AI 的总结代替验证。
2. 必查需求覆盖、依赖、变更范围、真实数据、错误路径、回归影响和文档同步。
3. T3 检查真实上下游；T4 从用户入口完整操作；T5 检查监控、回滚、安全和数据对账。
4. 结论只能是：通过、退回并列缺口、或阻塞并列外部条件。通过时记录验收人、时间、代码版本和证据位置。

### 提交规则

1. 一个任务一个独立提交或最小提交序列，提交信息必须含任务 ID。
2. 提交前核对实际文件未越界；禁止强推、改写他人提交、回滚未知改动、跳过钩子或改测试掩盖缺陷。
3. 公共契约先验收再供下游接入；冲突由主 Agent 指定处理人。
4. 文档、迁移、配置和测试属于功能交付；行为变化但文档未同步，不得进入“待验收”。
5. 合并到目标分支后必须复跑对应等级验证。
