# NovelCraft Personal Studio 技术实施方案（重制版）

## 1. 后端目录结构（V1 仓库演进）

```
backend/app/
├── api/v1/                  # auth, projects, contents, novels, runs, workflows,
│                            # knowledge, prompts, publish, metrics, admin
├── services/
│   ├── content_service.py        # C1：统一内容 + meta Schema 注册表
│   ├── version_service.py        # C5：VersionedRepository 封装
│   ├── workflow/                  # C2：引擎
│   │   ├── engine.py             # run 调度/节点状态机/断点续跑
│   │   ├── nodes.py              # agent/human/tool/branch 四类节点
│   │   └── presets/              # 八条预设工作流定义(YAML)
│   ├── narrative/                 # C6：叙事一致性引擎
│   │   ├── assembler.py          # 7层上下文装配（token预算制）
│   │   ├── summarizer.py         # 章/卷/全书分层摘要
│   │   ├── entity_state.py  foreshadow.py  timeline.py  arc.py
│   ├── knowledge/                 # C4：Knowledge Hub
│   │   ├── hub.py  ingest/(web,pdf,docx,md)  style_learn.py  book_analyze.py
│   ├── publish/                   # C7：发布网关
│   │   ├── base.py               # PublisherAdapter 抽象
│   │   ├── adapters/             # wechat, toutiao, xhs, zhihu, baijia, dayu,
│   │   │                         # wangyi, medium, substack, x, royalroad, kdp...
│   │   └── safety.py             # 敏感词/平台规则前置过滤
│   └── metrics_service.py         # 数据回流 + ROI
├── ai/                            # C3：AI Gateway
│   ├── gateway.py  budget.py  router.py
│   ├── providers/                 # deepseek, claude, openai, gemini（统一接口）
│   └── prompts/                   # 加载器 + builtin/*.yaml（seed 入库）
├── agents/                        # 10 个 AgentNode 定义（prompt+工具+Schema）
│   ├── producer.py story_architect.py character.py writer.py editor.py
│   ├── reviewer.py short_story.py social_media.py trend.py publisher.py
├── repositories/                  # base.py: ProjectScopedRepository(强制归属过滤)
├── workers/                       # celery_app(五队列) + beat(schedules表驱动)
├── models/ schemas/ core/ tests/
```

铁律：LLM 调用只准走 gateway（CI grep 守卫）；写操作只准走 VersionedRepository；内容读写只准走 content_service（meta Schema 校验）。

## 2. 前端结构与设计系统

```
frontend/src/
├── design/                # tokens.css(明暗双套) · primitives(Radix封装) · motion.ts
├── components/
│   ├── editor/            # Tiptap扩展：AI浮条/行内批注/版本树面板/diff视图
│   ├── workflow/          # RunTimeline · NodeCard · HumanGate · DAG编辑器(M2)
│   ├── novel/             # CharacterCard · OutlineTree · RadarScore · ForeshadowBoard
│   ├── studio/            # ContentMatrix · KnowledgeBrowser · TrendBoard
│   └── common/            # CommandK(⌘K) · DataTable · EmptyState ...
├── pages/                 # 按里程碑：wizard/progress/review/editor/costs →
│                          # serial-console/prompt-lab → matrix/knowledge/trends →
│                          # publish/analytics → settings/members
├── stores/                # zustand: auth/project/run(SSE)/editor/offline
├── offline/               # sw.ts(ServiceWorker) · db.ts(IndexedDB) · outbox.ts(任务出站队列)
├── api/                   # openapi-ts 生成客户端，类型与后端 schema 单一来源
└── hooks/                 # useSSE useAutosave useOffline useBudget useCommandK
```

- 状态：zustand（客户端态）+ TanStack Query（服务端态，离线时读 IndexedDB 持久化缓存）；
- 编辑器：Tiptap，文档格式统一 Tiptap JSON（= contents.body），派生输出时按平台序列化（MD/HTML/纯文本）；
- PWA：vite-plugin-pwa；L1 缓存策略 stale-while-revalidate；L2 编辑走本地草稿 + 版本树合并 UI；L3 outbox 在恢复网络后按序重放。

## 3. 数据库

### 3.1 建表顺序（依赖拓扑，M1 全建骨架）

```
users → projects → project_members → contents → derivations → versions
→ knowledge_items → knowledge_vectors(pgvector) → prompts → model_routes
→ workflows → schedules → workflow_runs → run_nodes → ai_calls → budgets
→ entity_states → foreshadowings → timeline_events → arcs → reviews
→ platform_accounts → publish_records → metrics → sensitive_words → audit_logs
```

### 3.2 关键设计
- contents.body 为 JSONB（Tiptap），全文检索用生成列 tsvector + GIN；章节按 (parent_id, meta->>'seq') 唯一索引；
- versions.snapshot JSONB + 大文本压缩（TOAST 默认足够，超 1MB 走 lo/外部对象再议）；auto_save 版本 beat 任务 7 天清理；
- knowledge_vectors 用 HNSW 索引；embedding 维度在 config 固化（换模型需迁移，明示）；
- ai_calls 按月分区（追踪表增长最快）；
- 迁移纪律：Alembic 线性单头、一 PR 一迁移、生产迁移自动前置备份、可降级。

### 3.3 初始化 seed
默认项目、八条预设工作流、内置 Prompt 集（含 model 分支与 golden cases）、默认 model_routes、敏感词基础词表、平台规则库初始条目。

## 4. AI 实施

### 4.1 模型路由与降级
model_routes：task_type → [primary(provider,model,params), fallbacks[]]，Redis 缓存热更新。降级链：主模型 429/5xx/超时 → 依序备胎 → 全败则 run_node 置 PENDING_PROVIDER 并告警（不是 FAILED，恢复后自动重试）。各 provider 独立令牌桶限流（复用 V1 Redis 限流件）。建议初始路由：草稿/摘要/字段生成 → DeepSeek；关键章节/重写/风格化 → Claude；审核 → 与生成异构（交叉审核）；出海翻译文学化 → Claude/GPT；Gemini 作长上下文备胎。

### 4.2 7 层上下文装配（narrative/assembler.py）
按架构评审 §7 层级与 token 配额组装，硬截断 + 低优先级先丢；装配结果与丢弃日志随 ai_calls.input 落库（防崩问题可回放定位"当时模型看到了什么"）。

### 4.3 Prompt 管理与实验
DB 为准（name+version+model 三键）；Jinja2 渲染；改动即新版本；每 prompt ≥3 条 golden case 进 CI；Prompt 实验室 = 批跑矩阵（同 input × prompt/model 组合）写 ai_calls + 对比视图；A/B = router 按权重分流 + 质量分/成本聚合报表。

### 4.4 Token 与成本控制
三级预算（Redis 计数 + 落库对账）：单任务/单项目/单日；输出 max_tokens = 目标字数×1.4；稳定上下文（世界观/人物卡/风格卡）启用 provider prompt caching；成本页从 ai_calls 聚合，无独立记账系统。

## 5. 部署运维（单 VPS）

compose 服务：nginx(TLS) / api / worker-plan / worker-generate / worker-review / worker-publish / worker-system / beat / postgres(pgvector) / redis。
- 资源基线：4C8G 起步，generate worker 并发=min(各 provider RPM 限额)；
- 备份：beat 每日 02:00 pg_dump | age 加密 → rclone 异地（对象存储），保留 30 天，每月恢复演练入日历；
- 健康与告警：/healthz 聚合（DB/Redis/队列深度/磁盘），异常与任务失败、预算触顶、provider 全降级 → Telegram Bot；
- 日志：structlog JSON → 本地滚动 + grep 够用（1~5 人规模不上 ELK）；
- 升级：镜像 tag 发布，`make deploy`（备份→pull→migrate→up→healthcheck→失败回滚上一 tag）。
