## 一、项目全局架构分析

**技术栈**：React 19 + TypeScript + Vite（前端）；FastAPI + 裸 psycopg2 连接池（无 ORM）+ PostgreSQL(pgvector) + Celery + Redis（后端）；DeepSeek 主 + Claude/OpenAI/Gemini 降级链（AI）；Docker Compose + Nginx（部署）；GitHub Actions（CI）。

**入口与分层**：`backend/app/main.py`（lifespan `init_db` 只做 seed，建表全靠 Alembic）；11 个 router 模块 + main 内联，共 129 条路由；`gateway.complete()` 为全站 AI 唯一入口（预算断言→路由→严格输出 Schema→ai_calls 账本）；`workers/tasks.py`（~950 行）承载 bootstrap 线性链、连续章节、批次、巡检 beat。

**关键调用链**：
- 扫榜成书：RankingCenter → `/ranking/sources/{s}/scan|import` → adapter/用户浏览器工件 → snapshots/items → `/analyze`(Gateway 严格 Schema) → topic_candidates → `/generate-book`(幂等键) → bootstrap 链 → contents(chapter)。
- 连续章节：BookLibrary → `/novels/{id}/continue`或`/chapters/batch` → 分布式锁 + 章节级 generation_key + `client_mutation_id` 重放 → 7 层上下文组装（book/volume/recent/entity/foreshadow/knowledge/outline）→ 连续性风险报告落 meta → 批次 `pending_provider`/resume 断点续跑。
- 发布：PublishDashboard → 敏感词前置检查 → `/publish` → publish_gateway（manual/semi/auto 三模式，海外 4 平台为 ManualAdapter 占位）→ publish_records。

**风险模块**：`workers/tasks.py` 超大文件强耦合；无 ORM（schema 与 SQL 双轨，本轮已两次实锤列名漂移）；`complete_api.py` 32 端点薄透传；DAG 编辑器保存的自定义图**无执行器**（现已显式 501，不再假跑）。

**死代码/占位现状**（本轮复核）：死组件仅剩 `PublishPage.tsx`（被 PublishDashboard 取代，未接线）；`agent_registry.py`（100 行）为声明式注册表，无独立执行体；海外发布 ManualAdapter 占位；`_local_embedding` 为 blake2b 哈希（显式标注非语义）。此前审计列出的 vector_search 假向量、AgentConsole 模拟数据、Settings 假统计、Review 假兜底、Studio 静态列表、协作页 404 路径、matrix "ready" 假执行、跨模型长度公式假评分、热点静默空成功、DagEditor 空 project_id、`import-chapters` 不落库——**已全部修复并有回归测试**。
