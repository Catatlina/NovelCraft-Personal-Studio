## 1. 前置依赖版本

严格对齐生产基线，避免「在我机器上能跑」。

| 组件 | 推荐版本 | 说明 |
|---|---|---|
| Python | 3.11.x（3.11.9+） | 后端，需 `<3.12` 以兼容部分依赖；FastAPI/SQLAlchemy 已验证 |
| Node.js | 20.x LTS（20.11+） | 前端构建 |
| pnpm | 9.x | 前端包管理（或 npm 10.x） |
| Docker | 26.x+ | 本地 pg/redis 栈 |
| Docker Compose | v2.27+ | `docker compose` 子命令 |
| PostgreSQL | 16 + pgvector 0.7.x | 本地可用 Docker 镜像 `pgvector/pgvector:pg16` |
| Redis | 7.x | 本地可用 Docker 镜像 `redis:7-alpine` |
| age | 1.2.x | 备份加密（仅运维本地需要） |
| rclone | 1.6x | 异地备份（仅运维本地需要） |
| Make | 任意 GNU Make 4.x | 命令封装 |

> macOS 用户可用 `brew install python@3.11 node@20 pnpm docker pgvector redis age rclone`。
> Windows 用户建议 WSL2 + Ubuntu 22.04，避免路径/信号问题。

## 2. 仓库结构总览与克隆

### 2.1 仓库布局（顶层）

```
novelcraft/
├── backend/                 # FastAPI 后端（Python）
│   ├── app/
│   │   ├── api/v1/          # 路由层
│   │   ├── services/        # 业务层（content/workflow/narrative/knowledge/publish/metrics）
│   │   ├── ai/              # C3 AI Gateway（gateway/router/budget/providers）
│   │   ├── agents/          # 10 个 AgentNode 定义
│   │   ├── repositories/    # 数据访问层（ProjectScopedRepository/VersionedRepository）
│   │   ├── workers/         # Celery 五队列 + beat
│   │   ├── models/ schemas/ core/ tests/
│   │   └── alembic/         # 迁移
│   ├── pyproject.toml
│   └── .env.example
├── frontend/                # React + Vite + TS（PWA）
│   ├── src/
│   │   ├── design/          # tokens.css · primitives · motion
│   │   ├── components/ editor/ workflow/ novel/ studio/ common/
│   │   ├── pages/ stores/ offline/ api/ hooks/
│   ├── package.json
│   └── vite.config.ts
├── docker-compose.yml       # 生产栈
├── docker-compose.dev.yml   # 本地开发栈
├── nginx/
├── scripts/                 # backup.sh / wait_health.sh / seed 等
├── Makefile
└── .env.example
```

### 2.2 克隆与分支

```bash
git clone git@github.com:your-org/novelcraft.git
cd novelcraft
git checkout develop          # 日常开发目标分支
# 新功能开分支：feat/task-0xx-简述
git checkout -b feat/task-012-novel-tree
```

分支策略回顾（详见《07-工程协作规范》§1）：
- `main` 生产，只收 release/hotfix，合入即 tag
- `develop` 集成目标
- `feat/*` / `fix/*` / `hotfix/*` 功能/缺陷/急修

## 3. 后端本地搭建

### 3.1 虚拟环境与依赖

推荐 Poetry（仓库已带 `pyproject.toml`）；无 Poetry 可用 venv + pip。

```bash
cd backend
# 方式 A：Poetry
poetry env use python3.11
poetry install

# 方式 B：venv + pip
python3.11 -m venv .venv
source .venv/bin/activate
pip install -U pip
pip install -e ".[dev]"        # 含 ruff/mypy/pytest 等开发依赖
```

### 3.2 .env（本地）

复制示例并填本地值：

```bash
cp .env.example .env
# 编辑器填：DB/Redis/JWT/FERNET/Provider Keys；AI 功能必须配置真实 provider key
```

本地 `.env` 关键项（值与生产同源结构）：

```bash
APP_ENV=development
DB_HOST=localhost
DB_PORT=5432
DB_NAME=novelcraft_dev
DB_USER=nc_user
DB_PASSWORD=local_dev_pwd
REDIS_HOST=localhost
REDIS_PORT=6379
REDIS_PASSWORD=
JWT_SECRET=dev_only_not_secret
FERNET_KEY=<python -c "from cryptography.fernet import Fernet;print(Fernet.generate_key().decode())">
DEEPSEEK_API_KEY=<本地开发 key；不得提交>
# ... 其余 provider 可留空；未配置的 provider 调用会直接报错
```

### 3.3 本地 Postgres + Redis（Docker）

```bash
# 启动依赖（pgvector + redis）
docker compose -f docker-compose.dev.yml up -d postgres redis

# 建库（首次）
docker compose -f docker-compose.dev.yml exec postgres \
  psql -U nc_user -c "CREATE DATABASE novelcraft_dev;"
```

### 3.4 Alembic 迁移 + Seed

```bash
cd backend
# 迁移到最新
alembic upgrade head

# 初始化种子（默认项目/8 条预设工作流/内置 Prompt/敏感词/平台规则）
python -m app.seed
```

### 3.5 启动 API（uvicorn）

```bash
# 开发热重载
uvicorn app.main:app --reload --port 8000 --host 0.0.0.0

# 验证
curl http://localhost:8000/healthz
# => {"status":"ok", ...}
```

### 3.6 启动 Celery Worker + Beat

开多个终端（或用 tmux）：

```bash
# worker（五队列可分别或合一）
celery -A workers.celery_app worker -Q plan,generate,review,publish,system -c 4 -l info

# beat（定时调度，读取 schedules 表）
celery -A workers.celery_app beat -S redbeat.RedBeatScheduler -l info
```

> 本地调试可只起一个合并 worker；生产严格按队列分服务（见部署手册 §1.2）。

## 4. 前端本地搭建

### 4.1 安装依赖

```bash
cd frontend
pnpm install          # 或 npm install
```

### 4.2 由 OpenAPI 生成类型安全的 API 客户端

后端 schema 为**单一来源**，前端用 `openapi-ts` 生成：

```bash
# 先确保后端 /openapi.json 可达
pnpm openapi-ts       # 读取 http://localhost:8000/openapi.json -> src/api/
```

生成的客户端位于 `src/api/`，严禁手改；后端改 schema 后重跑本命令。

### 4.3 Vite Dev + PWA 本地调试

```bash
pnpm dev              # Vite dev server，默认 http://localhost:5173
```

PWA 本地调试注意：
- Service Worker 在 dev 下默认不注册（vite-plugin-pwa `devOptions.enabled=false` 默认）；
- 要测离线能力，临时 `devOptions.enabled=true` 并 `pnpm build && pnpm preview`；
- Lighthouse PWA 评分在 preview 模式下验证。

### 4.4 前端代理

`vite.config.ts` 已配 `/api` 代理到 `localhost:8000`，无需额外 CORS 配置（生产由 Nginx 统一）。

## 5. 一键起本地栈（docker-compose.dev.yml）

不想手动起 pg/redis，一条命令拉起「数据依赖 + 可选后端/前端」：

```bash
docker compose -f docker-compose.dev.yml up -d
```

`docker-compose.dev.yml` 示例（仅依赖 + 热挂载后端，前端仍本地 `pnpm dev` 便于 HMR）：

```yaml
# docker-compose.dev.yml
services:
  postgres:
    image: pgvector/pgvector:pg16
    environment:
      POSTGRES_DB: novelcraft_dev
      POSTGRES_USER: nc_user
      POSTGRES_PASSWORD: local_dev_pwd
    ports: ["5432:5432"]
    volumes:
      - pg_dev:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U nc_user"]
      interval: 5s
      retries: 10

  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      retries: 10

volumes:
  pg_dev:
```

> 全栈容器化（含 api/worker）可选：在 dev compose 中加 `api`/`worker` 服务并挂载源码 `--reload`。但前端 HMR 与后端重载在容器内容易卡，新手建议「容器只跑 pg/redis，前后端本地进程」。

## 6. LLM Provider 本地配置

### 6.1 用 DeepSeek 起步（推荐）

注册 DeepSeek 拿 key，填入 `.env` 的 `DEEPSEEK_API_KEY`，启动后端即可。初始路由（见《05》§4.1）：
- 草稿/摘要/字段生成 → DeepSeek
- 关键章节/重写/风格化 → Claude
- 审核 → 异构（与生成不同 provider）

### 6.2 无 mock 模式

当前产品规则：不再提供 Gateway mock 适配器。开发期无额度或 key 缺失时，AI 相关接口必须直接失败并返回明确错误；不得返回结构化占位内容、固定字典或“估算”产物。

```bash
# 本地真实 AI 冒烟示例
set -a
source .env.local
set +a
cd backend
.venv/bin/pytest tests/test_real_provider_v2_bootstrap.py -q
```

测试可以 monkeypatch 外部 HTTP 边界以覆盖失败路径，但不能把 mock provider 暴露成产品能力，也不能把录制/伪造输出作为真实验收证据。

### 6.3 Provider 配置热更新（model_routes）

`model_routes` 表存 task_type → [primary, fallbacks]。本地改路由无需重启：

```bash
# 改 DB 后清 Redis 缓存即可生效
redis-cli DEL "model_routes:*"      # 或等缓存 TTL
```

降级链：主 429/5xx/超时 → 备胎 → 全败置 `PENDING_PROVIDER` 并告警（开发期会打日志）。

## 7. 常用开发命令速查（make）

`Makefile` 顶层封装，避免记一长串参数。

```bash
# 后端
make backend-install     # poetry/venv 安装依赖
make migrate             # alembic upgrade head
make migrate-down        # alembic downgrade -1
make seed                # 初始化种子
make test                # pytest（需 pg+redis 服务容器）
make test-cov            # pytest --cov
make lint                # ruff check .
make type                # mypy app
make format              # ruff format .
make worker              # 起合并 worker（本地调试）
make beat                # 起 beat

# 前端
make fe-install          # pnpm install
make fe-dev              # pnpm dev
make fe-build            # pnpm build
make fe-openapi          # openapi-ts 生成客户端
make fe-lint             # eslint + stylelint + tsc

# 全栈
make dev-up              # docker compose -f docker-compose.dev.yml up -d
make dev-down            # ... down
make deploy TAG=v1.2.3   # 生产升级（见部署手册 §6）
```

> CI 对应门禁：ruff + mypy + tsc + stylelint(tokens 守卫) + 单测(pg+redis) + prompt golden case + LLM 直连 grep 守卫 + docker build。

## 8. IDE 配置建议

### 8.1 后端（VS Code / PyCharm）

- **ruff**：启用 `ruff` 扩展做格式化 + lint，保存即 fix。
  ```json
  // .vscode/settings.json
  {
    "python.defaultInterpreterPath": "./backend/.venv/bin/python",
    "[python]": { "editor.formatOnSave": true, "editor.defaultFormatter": "charliermarsh.ruff" },
    "ruff.importStrategy": "fromEnvironment",
    "mypy-type-checker.args": ["--strict"]
  }
  ```
- **mypy**：项目根 `pyproject.toml` 已配 `strict = true`，类型注解全覆盖。
- **pre-commit**：
  ```yaml
  # .pre-commit-config.yaml
  repos:
    - repo: https://github.com/astral-sh/ruff-pre-commit
      rev: v0.5.0
      hooks: [ { id: ruff, args: [--fix] }, { id: ruff-format } ]
    - repo: https://github.com/pre-commit/mirrors-mypy
      rev: v1.10.0
      hooks: [ { id: mypy, args: [--strict, app] } ]
  ```
  安装：`pre-commit install`。

### 8.2 前端（VS Code）

- **ESLint + Prettier + Stylelint**：
  ```json
  {
    "editor.defaultFormatter": "esbenp.prettier-vscode",
    "editor.codeActionsOnSave": { "source.fixAll.eslint": "explicit", "source.fixAll.stylelint": "explicit" },
    "stylelint.validate": ["css"],
    "eslint.validate": ["typescript"]
  }
  ```
- **stylelint 守卫**：严禁裸色值/魔法间距，只允许 `design/tokens.css` 的变量（CI 拦截）。
- **TS strict**：`tsconfig.json` 开 `strict: true`，禁止 `any`（CI tsc 报错）。

### 8.3 通用

- 推荐 VS Code 插件：Docker、REST Client（调 `/openapi.json`）、GitLens、Error Lens。
- 终端复用：tmux 起 api / worker / beat / vite 四窗。

## 10. 常见问题 FAQ

### Q1：迁移报 `relation already exists`
A：开发库被手动改过。可 `alembic downgrade base` 后 `upgrade head` 重建（开发库允许）。生产**严禁**这样操作。

### Q2：uvicorn 启动报 `ModuleNotFoundError: app`
A：没在 `backend/` 目录运行，或 venv 未激活。确认 `pwd` 与 `which python`。

### Q3：worker 连不上 Redis
A：Redis 没起或密码不对。先 `docker compose -f docker-compose.dev.yml ps` + `redis-cli ping`。`.env` 的 `REDIS_PASSWORD` 与 compose 一致（本地常留空）。

### Q4：pgvector 扩展创建失败
A：用的是普通 postgres 镜像而非 `pgvector/pgvector:pg16`。换镜像，或在已有库执行 `CREATE EXTENSION IF NOT EXISTS vector;` 失败即镜像不支持。

### Q5：openapi-ts 生成报错 DNS
A：后端未起或 `/openapi.json` 不可达。先 `curl localhost:8000/openapi.json` 验证，再跑 `pnpm openapi-ts`。

### Q6：前端 SSE 进度不更新
A：本地 dev 用 Vite 代理，SSE 正常；若自搭 Nginx 才需关缓冲（见部署手册 §4）。dev 下确认 `pnpm dev` 控制台无代理报错。

### Q7：golden case 测试要真实 key？
A：是的。golden case 必须真实 provider 跑（CI 真实额度）。本地无 key 时 AI 相关 golden 会 skip 或失败；不得启用 mock provider 代替真实验收。

### Q8：改了 model_routes 不生效
A：Redis 缓存未清。本地 `redis-cli DEL "model_routes:*"` 或等 TTL。

### Q9：stylelint 报「Unexpected value」裸色值
A：组件里用了 `#fff` 之类。改用 `var(--color-...)`（见 `design/tokens.css`）。CI 会拦。

### Q10：pre-commit 卡住 / 慢
A：首次 mypy 全量慢属正常；可加 `SKIP=mypy` 临时跳过，但提交前务必本地过一遍 type。

### Q11：Celery 任务不消费
A：`-Q` 队列名拼错或 beat 没派。检查 `celery inspect active` 与 `model_routes`/schedules 表。

### Q12：想重置整个本地库
A：`docker compose -f docker-compose.dev.yml down -v` 删卷 → 重新 `up` → `alembic upgrade head` → `seed`。
