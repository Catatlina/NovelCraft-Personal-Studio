# NovelCraft 开源项目融合基线

> 版本：V1.1 ｜ 日期：2026-07-11 ｜ 状态：六个用户指定项目均纳入能力融合范围
>
> 本文是功能设计、实现评审和验收的强制输入。六个项目的目标能力都要融合进 NovelCraft；“不复制代码”仅是许可证隔离方式，不代表不实现其能力。

## 1. 融合原则

1. 先固定上游 commit，再分析代码、Prompt、数据结构和测试，禁止只看 README 或 Stars。
2. MIT / Apache-2.0 项目可在保留许可证、版权声明和 NOTICE 要求的前提下直接移植或改写；AGPL-3.0 / GPL-3.0 项目的目标能力采用**洁净室独立实现**：先编写行为规格和验收测试，再独立完成等价能力。除非项目整体许可证另行决策，不把其源代码、派生代码或原 Prompt 文本并入仓库。
3. 参考不等于整仓复制。NovelCraft 保留 FastAPI + PostgreSQL + Redis + Celery + React 架构，只移植可验证的领域协议、Prompt、状态机和测试思想。
4. 不导入参考项目附带的受版权保护小说正文；扫榜只采集合法公开元数据，拆文只处理用户有权使用的文本。
5. 榜单采集基础设施采用 Playwright；任务队列复杂后可评估 Apache-2.0 的 Crawlee；番茄渲染文字识别可采用 Apache-2.0 的 PaddleOCR。三者仅承担浏览器、调度和 OCR 能力，不用于验证码破解、签名破解或访问控制绕过。
6. 小说下载器、盗版书库及其接口不进入运行时依赖或校验源。可研究公开的适配器边界，但不得复制站点破解代码、下载正文或把不可授权数据写入知识库。

### 榜单采集组件边界

| 组件 | 用途 | NovelCraft 接入方式 | 禁止行为 |
|---|---|---|---|
| Microsoft Playwright | 可见浏览器、持久会话、截图与公开 DOM 提取 | `BrowserCaptureAdapter` 工件协议 | 自动过验证码、隐藏绕过挑战 |
| Apify Crawlee | 队列、重试、会话与采集调度 | 数据量扩大后可选，不作为首期硬依赖 | 代理轮换规避封禁 |
| PaddleOCR | 番茄榜单卡片渲染文字识别 | 独立 OCR worker，输出坐标和置信度 | OCR 小说正文或批量还原受保护内容 |
| Open Library | 书名、作者、ISBN 等开放目录交叉校验 | `MetadataValidator`，失败不阻断原始快照 | 将“未收录”判定为原数据错误 |
| CSV/JSON | 用户授权数据和故障兜底 | 严格字段校验、来源和行号留痕 | 导入来历不明的小说全文 |
5. 每项融合必须新增来源标记、差异说明、失败路径测试和端到端验收。

### 2.1 oh-story-claudecode 融合清单

- 以 `story-long-scan` 为榜单中心第一参考：起点、番茄、晋江、七猫、刺猬猫 adapter 独立封装，输出统一快照格式。
- 以 `scan-output-format`、`topic-decision`、`genre-trends` 形成“榜单条目→市场卡→原创选题卡”，不得把榜单作品正文直接送入成书 Prompt。
- 以 `story-long-analyze` 的概要/章节/角色/剧情/设定协议映射到 Knowledge Hub 的 `analysis/style_card/character/worldview`。
- 将 `story-deslop` 的确定性规则拆成版本化规则集；阻断规则与建议规则分级，AI润色前后均执行。
- 将番茄/起点/知乎 rubric 变成结构化 Reviewer Schema，禁止无正文默认给分。
- “33个 Prompt”按用途进入 Prompt Registry，逐个补变量契约、输出 Schema 和至少3个 golden case；不能把 Skill 原文直接拼成一个超长 Prompt。

### 2.2 denova 融合清单

- 引入 `WorkflowPlan` 数据协议，Producer只产计划，执行器只执行持久化 DAG。
- run必须有 checkpoint、事件 ledger、节点输入输出、mutation记录和 post-run verifier。
- Agent、Skill、Tool分别注册；节点只引用注册ID，不允许前端自由字符串绕过契约。
- 自动化触发器统一承载榜单扫描、热点扫描、自动连载、定时发布；触发与执行分离且幂等。
- SSE采用结构化事件链，支持游标续传；Agent Console展示 trace、token、工具调用和失败原因。
- 只融合可控的显式工作流，不引入无限自治对话。

### 2.3 show-me-the-story 融合清单

- 书库项目页覆盖 Projects、Outline、Writing、Foreshadows、Memory、Relations 等核心视图。
- 章节生成采用“细纲→正文→长度校验→审稿→伏笔/事实一致性→后处理→落库”事务链。
- 伏笔支持新增、预计回收章、状态转换、冲突检查和回收验证，不是仅有一张查询表。
- 章节事实与记忆采用可查询的有界上下文，写作前装配、写作后 reconcile。
- 引入正文长度、截断、引号、元数据和删除恢复等边界测试。

## 3. 第二批能力融合项目

| 项目 | 核验基线 | 许可证 | 必须融合的能力 | 实现方式 |
|---|---|---|---|---|
| [YILING0013/AI_NovelGenerator](https://github.com/YILING0013/AI_NovelGenerator) | `f9aefef` | AGPL-3.0 | 多章节衔接、章节目录解析、向量知识库、伏笔与一致性检查、架构/蓝图/章节/定稿分层 | 洁净室独立实现，融合行为与数据协议，不复制 AGPL 代码/Prompt |
| [wfcz10086/AI-automatically-generates-novels](https://github.com/wfcz10086/AI-automatically-generates-novels) | `5975df2` | Apache-2.0 | 批量生产、智能拆书、Prompt可视编辑、快捷词条、书名简介、思维导图、多模型入口 | 可直接移植适配或重写；保留 Apache-2.0 归属与变更记录 |
| [XTmingyue/harnessNovel](https://github.com/XTmingyue/harnessNovel) | `0d95508` | GPL-3.0 | 世界知识抽取/合并、故事弧、长主线、虚拟卷切分、卷纲/章纲、自适应草稿、合理性审计、humanize | 洁净室独立实现，融合流水线职责和验收行为，不复制 GPL 代码/Prompt |

### 3.1 AI_NovelGenerator 能力落点

- `chapter_directory_parser` → `ChapterDirectoryImport`：识别多种章节标题、卷章层级和缺号，导入后进入统一书库。
- architecture/blueprint/chapter/finalization 四阶段 → 扫榜成书工作流的设定、蓝图、章节批次、定稿节点。
- knowledge/vectorstore/embedding → Knowledge Hub 项目级长期记忆；写前检索、写后事实提取与回写成对执行。
- consistency checker → Reviewer 的人物、地点、时间、物品、设定与伏笔六类一致性报告，可定位章节并触发返工。
- 验收重点：连续 100 章上下文不无限膨胀、目录导入不乱序、重跑不重复写入知识和章节。

### 3.2 AI-automatically-generates-novels 能力落点

- book splitter → 拆书工作台：分卷/分章、批处理、结构卡入库，禁止把无权使用的正文带入生成端。
- prompt editor/final preview → Prompt 实验室：变量提示、最终渲染预览、版本对比、回滚和 golden case 批跑。
- mode shortcut → 编辑器命令面板：用户自定义快捷词条、项目级模板和快捷润色操作。
- bookinfo/name/synopsis → 选题卡到书名、卖点、简介的多候选生成与评分。
- mind map → 大纲/人物关系/故事弧只读可视化；权威数据仍在结构化数据库。
- 多模型入口只接入现有 AI Gateway，不复制各模型 app 的直连方式。

### 3.3 harnessNovel 能力落点

- 拆书输出分为 worldview、story pattern、story arc、world canon index 和 rewrite map，作为市场结构知识卡，不保存可识别原文表达。
- 新书链增加“核心玩法→新世界观→长主线→阶段路线→故事弧→虚拟分卷→卷纲→连续章纲”的分层规划。
- 写作链增加 world knowledge extract/merge、arc context、batch summary、reasonability audit、supplement usage audit 和 humanize 后处理。
- 自适应草稿与连续章纲采用窗口化生成：每批有 checkpoint、预算、输入摘要和写后 reconcile，支持暂停后继续。
- 所有 Prompt 依据公开功能职责重新编写，并通过行为验收，不复制 GPL Prompt 文本。

## 4. 融合后的核心工作流

### 4.1 扫榜全自动成书

`RankScan → Snapshot → MarketAnalysis → OriginalTopic → BookCreate → World/Character → Outline/Volume/ChapterOutline → ChapterBatch → Review → Rewrite → Reconcile → Library`

- `BookCreate`一执行，小说立即在书库可见，状态为 `planning/generating`；失败时保留项目和失败节点，不丢书。
- 自动模式无需用户逐节点点击，但质量、安全、预算或数据源异常时进入可恢复等待态。
- 每次成书记录榜单快照、分析卡、Prompt版本、模型、工作流版本和参考协议版本。

### 4.2 热点全自动自媒体

`HotspotScan → Deduplicate → Value/Lifecycle → PlatformFit → TopicPool → MultiPlatformDraft → Review → PublishQueue → Metrics → Optimization`

## 5. 融合验收门禁

- 扫榜 adapter 至少2个真实源 + 1个 fixture 源，全都输出统一 Schema；失败可见并可手动重放。
- 从榜单选择到小说出现在书库不超过一次工作流提交；完整来源链可追溯。
- 20章自动生成测试中，人物、伏笔、时间线和章节事实均有写后 reconcile 记录。
- 去AI规则同时有确定性 lint 与模型审查，规则版本可回放。
- Workflow kill/restart后从 checkpoint 恢复，不重复创建书或章节、不重复扣费。
- 对GPL/AGPL参考项目执行独立实现证明：设计说明、无代码复制审查和差异测试归档。
- 六个项目分别维护 `upstream_manifest`：仓库、commit、许可证、能力映射、采用方式、实现文件、测试编号和差异说明；没有清单的能力不得声称已融合。
- 第二批门禁：目录导入 fixture、100章记忆压力测试、拆书批处理、Prompt预览回滚、快捷词条、分层规划、知识合并、合理性审计和 humanize 对照测试全部通过。

## 6. 后续更新要求

每次上游升级先记录新 commit 和变更摘要，再决定是否同步；禁止直接跟随 `main/master` 构建生产版本。
