## 1. 总体架构（1~5 人规模定型）

```
React + Vite + TS（PWA，设计系统，Tiptap 三栏工作台）
        │  REST + SSE（进度/通知）
FastAPI（api → service → repository，JWT + Owner/Editor/Viewer）
        │
├── PostgreSQL + pgvector   唯一事实源（内容/知识/版本/追踪/向量）
├── Redis                   Celery broker · 限流 · 预算计数 · 缓存 · SSE 发布
├── Celery + beat           五队列（plan/generate/review/publish/system）+ 定时自动化
├── AI Gateway（进程内模块） 4 家 Provider 路由/降级/预算/Schema 校验/全量追踪
└── Nginx + HTTPS · Docker Compose · 单 VPS
```

**结论：✅ 通过。** 不引入任何新基础设施组件。工作流引擎、发布网关、Knowledge Hub 全部是 FastAPI 进程内模块 + Celery 任务，维护成本恒定。

## 2. C1 统一内容模型（Everything is Content）——全系统第一设计决策

```sql
contents(
  id, project_id, parent_id,          -- parent_id: 章节属于书、派生稿指向源稿
  type,          -- novel/volume/chapter/short_story/flash_fiction/
                 -- wechat_article/toutiao/xhs_note/zhihu_answer/video_script/...
  title, body,   -- body: Tiptap JSON（统一编辑格式）
  meta JSONB,    -- 类型专属字段：章节序号/平台标签/SEO标题/封面文案/分镜…
  status, owner_id, created_at, updated_at
)
derivations(source_content_id, derived_content_id, workflow_run_id)  -- 复用血缘
```

- 小说 = content(type=novel) + 子树（volume/chapter 都是 content），**书籍结构即内容树**；
- "一稿多平台" = 对同一 source 的 fan-out 派生，血缘可追溯；
- 新平台/新体裁 = 新增一个 type 枚举 + meta schema + prompt 模板，**零表结构变更**。

⚠️ 评审警示：meta JSONB 是灵活性来源也是失控来源——每个 type 的 meta 必须有 Pydantic Schema 注册表校验，禁止自由写入。

## 3. C2 工作流引擎（Automation First）——第二骨架

清单中"预设/自定义工作流、节点开关/排序、模型选择、人工确认、暂停/恢复/取消/重试、定时/批量、自动连载、自动发布"全部由此承载。

```sql
workflows(id, name, is_preset, definition JSONB)   -- definition: 节点数组(DAG)
workflow_runs(id, workflow_id, project_id, status, context JSONB, schedule_id)
run_nodes(id, run_id, node_key, agent, status,     -- pending/running/waiting_human/
          input JSONB, output JSONB, ai_call_ids,  -- succeeded/failed/skipped
          attempt, started_at, finished_at)
schedules(id, workflow_id, cron, enabled)          -- Celery beat 读取
```

节点类型仅四种（保持引擎极简）：**agent 节点**（调 AI）、**human 节点**（挂起等确认，SSE + 通知）、**tool 节点**（抓热点/发布/翻译等确定性操作）、**branch 节点**（按条件分流，如质量分<80 → 返工）。

执行语义（验收红线）：节点级幂等（重跑不产生脏数据）、run 级断点续跑、human 节点挂起不占用 worker、失败节点单独重试不重跑全链、全部输入输出落 run_nodes（与 C8 追踪打通）。

预设工作流首批八条：**扫榜全自动成书（V2.2 首要）** / 热点自媒体矩阵（V2.2 次主链）/ 长篇章节流水线 / 黄金三章 / 短篇爆款 / 自动连载 / 出海翻译发布 / 拆书入库。灵感入口复用扫榜成书后半段节点，不维护独立小说引擎。

## 4. C3 AI Gateway + Agent 层

### 4.1 Gateway
统一入口 `gateway.complete(task_type, prompt_name, variables, output_schema, budget_scope)`：
1. 预算检查（任务/项目/日三级，Redis 计数）→ 2. 路由（model_routes 表：task_type → provider/model/参数，热更新）→ 3. Prompt 渲染（带 version + model 分支）→ 4. Provider 调用（限流令牌桶、指数退避重试、**降级链**：主模型失败→备用模型→入队等待）→ 5. Schema 校验（失败带错误重试 ≤2）→ 6. 全量落库 `ai_calls`。

```sql
ai_calls(id, run_node_id, agent, task_type, provider, model, prompt_name,
         prompt_version, input JSONB, output JSONB, prompt_tokens,
         completion_tokens, cost, latency_ms, status, error, created_at)
```
这一张表同时支撑：Agent 输入输出记录、成本统计（单章/单篇/单模型）、模型质量对比、A/B 测试数据、Prompt 实验室回放。**Traceable 不是独立系统，是这张表。**

### 4.2 Agent 形态（✅ 定案，原 ⚠️ 已关闭 → 见 §13.1）
10 个 Agent（Producer/Story Architect/Character/Writer/Editor/Reviewer/Short Story/Social Media/Trend/Publisher）**不做自治对话式多 Agent**，统一实现为：

> `AgentNode = 角色 SystemPrompt + 任务 Prompt + 工具白名单 + 输出 Schema`，由工作流引擎显式编排。

"多 Agent 协作" = 工作流 DAG；"Agent 状态追踪" = run_nodes；可测试、可重放、可断点。Producer Agent 是唯一特殊者：它是工作流的**规划者**（把用户目标翻译为一次 workflow_run 的参数），不是运行时指挥官。

**定案补强（本版新增，落实原 ⚠️ 修正项）**：

1. **Agent 契约固化**：每个 AgentNode 在 `agents/` 下声明 `AgentSpec = {role_prompt, task_prompt_name, tool_whitelist[], input_schema, output_schema, max_retries, budget_scope}`，由 `agent_registry` 统一注册；引擎只能通过 registry 实例化 Agent，禁止 Agent 之间直接互调（CI 静态检查：`agents/**` 不得 import 其它 `agents/**`）。
2. **Producer 的边界**：Producer 只输出 `WorkflowPlan`（选哪条预设/如何填参/human 节点插入点），产物是**数据**不是控制流；Plan 交给引擎执行，Producer 不参与运行期。Plan 本身进 ai_calls 可回放。
3. **禁自治的机器约束**：工具白名单默认不含"调用其它 Agent"这一工具；任何 Agent 想触发另一 Agent，必须由引擎按 DAG 边推进。→ 从代码层面杜绝"自治对话循环"。
4. **可测试红线**：每个 AgentNode 必须有 ≥1 条 golden case（固定 input → 断言 output schema + 关键字段），进 CI。

### 4.3 Prompt 实验室与 A/B
prompts 表（name, version, model, template, output_schema, changelog, golden_cases）。实验室 = 同 input × {prompt 版本 / 模型} 矩阵批跑 + ai_calls 对比视图；A/B = model_routes 支持按百分比分流，效果看 ai_calls 聚合 + 质量分。

## 5. C4 Knowledge Hub（Everything is Knowledge）

```sql
knowledge_items(id, scope,            -- global / project_id
  kind,   -- character/worldview/setting/hotspot/article/golden_line/title/
          -- platform_rule/brand_style/style_card/prompt_ref/webpage/file
  title, content, meta JSONB, source_url, file_ref, version_head)
knowledge_vectors(item_id, chunk_no, embedding vector, chunk_text)
```

- 入库通道：手动录入、网页抓取、PDF/Word/Markdown 解析（Celery 任务）、小说原文自动切片、热点采集器写入、**拆书分析产物**（开篇结构/爽点/节奏/标签 → 结构化 style/analysis 卡）；
- RAG：统一检索接口 `kb.search(query, scope, kinds, k)`，7 层上下文装配的"素材层"从这里取；

### 5.1 风格学习 / 仿写（✅ 定案，原 ⚠️ 已关闭 → 见 §13.4）
风格学习 = 样本集 → 特征提取 Agent → style_card（叙事人称/句长分布/高频意象/节奏特征）→ 生成时注入。**定案的防侵权红线（三道闸）**：

1. **入库闸**：样本入库时记录 `source_type`（原创/授权/公共领域/第三方）与授权声明；第三方受版权保护样本只允许抽取**统计特征**（句长/词频/节奏），禁止整段原文进 style_card 的可注入字段。
2. **生成闸**：仿写输出强制过相似度检测——`sim = max(向量余弦, 归一化 5-gram 重合率)`；`sim ≥ 0.75` 触发强制重写（最多 2 轮），`0.6 ≤ sim < 0.75` 标记高风险并要求 human 节点确认，`sim < 0.6` 放行。阈值写入 config 可调，检测记录落 ai_calls.meta。
3. **产物闸**：任何进入发布网关的仿写产物必须携带"相似度检测通过"标记，否则发布网关拒绝（与 §8 内容安全过滤同一拦截点）。

红线验收：构造与样本高度雷同的输入，检测器必须拦截并触发重写；重写后 sim 必须下降到阈值以下方可放行。

## 6. C5 通用版本系统（Everything is Versioned）

```sql
versions(id, entity_type,      -- content/knowledge_item/prompt/workflow
         entity_id, version_no, parent_version_id,   -- parent 形成版本树
         snapshot JSONB, reason,                     -- manual/ai_rewrite/auto_save/restore
         author_id, created_at)
```
- 写路径统一走 `VersionedRepository.save()`：先快照后更新，任何实体自动获得版本能力；
- 正文类 diff 用文本 diff（diff-match-patch），结构化实体用字段级 diff；
- "内容版本树"：AI 重写产生分支而非覆盖，编辑器提供树视图 + 任意节点恢复；
- 保留策略：auto_save 版本 7 天滚动清理，manual/ai_rewrite 永久。

## 7. C6 叙事一致性引擎（护城河，小说域专属）

在 Content/Knowledge 之上叠加小说专属结构：

- **7 层上下文装配**（token 预算制，超限按优先级从低到高丢弃）：
  ① 全书状态摘要 ≤1k → ② 本卷摘要 ≤800 → ③ 近 3 章摘要 ≤600 → ④ 实体状态表切片 ≤500 → ⑤ 到期伏笔/时间线提醒 ≤300 → ⑥ RAG 召回素材 ≤1.5k → ⑦ 本章细纲 + 写作规则 ≤700；
- **实体状态表** entity_states（人物位置/关系/持有物/已知信息，随章更新，硬约束注入）；
- **伏笔** foreshadowings（埋设章/内容/计划回收章/状态，到期强制提醒 + Reviewer 校验回收）；
- **时间线** timeline_events（章内事件抽取入表，跨章矛盾检测）；
- **人物弧线** arcs（阶段目标 → Reviewer 按卷校验偏移）；
- **检测族**（OOC/设定冲突/前文一致性/节奏）= Reviewer Agent 的检查维度，与 7 维质量分合并输出；
- **防崩总策略**：章纲先审（便宜 20 倍）→ 成稿交叉模型审（生成与审核用不同 provider）→ 每 10 章巡检 → 卷级人工复盘门禁。目标：人工修订率 <10%，不承诺零。

去AI味/润色/扩写/缩写/改写 = Editor Agent 的操作集，编辑器选中即调（走 Gateway，同样追踪计费）。

## 8. C7 发布网关 + 数据回流

- 适配器模式 `PublisherAdapter`（format() / publish() / fetch_metrics()），每平台一个 adapter；
- 数据回流分级：API 平台自动拉取 → 无 API 平台半自动（粘贴/截图 OCR 录入）→ 手动录入兜底；metrics 表按 content×platform×date 聚合，供表现分析与 ROI 面板。

### 8.1 自动发布三模式（✅ 定案，原 ⚠️ 已关闭 → 见 §13.2）
发布前强制过内容安全过滤（敏感词表 + 平台规则库校验 + §5.1 仿写相似度标记）。**发布三模式定案**：

| 模式 | 适用平台 | 触发方式 | 风险控制 |
|---|---|---|---|
| 手动 | 全部 | 排版 + 一键复制，人工去平台粘贴发布 | 零自动化风险 |
| 半自动（默认） | 公众号/头条/小红书/知乎/百家/大鱼/网易 等 | Playwright 填充内容，**人点最终"发布"按钮** | 不代替人做发布动作 |
| 全自动 | 仅官方 API 平台：Medium / Substack / X API / WordPress | 系统直接调用官方 API | 官方授权，合规 |

**国内平台全自动的强约束（定案）**：
1. 默认关闭；开启需在设置页勾选"我已知悉平台条款与封号风险"并二次确认弹窗，操作写 audit_logs；
2. 全自动工作流在"发布"tool 节点前**强制插入 human 确认节点**（首次开启后可按平台记住选择，但每平台首篇仍需人工确认）；
3. 每次全自动发布落 publish_records 并附风险提示副本；连续失败（如触发验证码/风控）自动降级为半自动并告警。

红线验收：国内平台在未显式开启全自动时，发布路径只能落到半自动（Playwright 填充后停在待人工点击态）；开启后首篇仍出现 human 确认节点。

## 9. C7-前端 / 离线（Offline First 三级）（✅ 定案，原 ⚠️ 已关闭 → 见 §13.3）

- **设计系统先行**（MVP 前 1.5~2 周）：design tokens（色彩/字号/间距/圆角/阴影）、暗色模式原生、组件库（基于 Radix/shadcn 定制）、动效规范（Framer Motion，克制使用）；对标基准：信息密度学 Linear、编辑沉浸学 Notion/Cursor、命令交互学 Raycast（⌘K 全局命令面板列入 P1）；
- **AI 无处不在**的交互标准：全局 ⌘K、编辑器选中浮条（续写/改写/润色/扩缩写）、右键 AI 菜单、实体卡片 "AI 补全" 按钮、发布页 "AI 优化标题"——全部是同一个 Gateway 的薄调用，UI 成本为主。

**Offline First 三级定案（不做离线自动合并与实时协同）**：

| 级别 | 能力 | 技术实现 | 冲突策略 |
|---|---|---|---|
| L1 只读 | 项目树/章节/人物/知识/版本可浏览、可搜索 | Service Worker（stale-while-revalidate）+ IndexedDB 缓存 | 无写入，无冲突 |
| L2 离线编辑 | 断网可编辑并本地暂存，恢复网络自动提交 | IndexedDB 本地草稿 + `base_version_id` 乐观锁 | 提交时若服务端版本已变 → 生成分支进版本树，弹**三方对比 UI**由人选择，绝不静默覆盖 |
| L3 AI 出站队列 | 离线创建的 AI 任务排队，恢复后按序自动执行 | `outbox` 队列（IndexedDB）+ 恢复网络后重放，幂等键防重复 | 依赖服务端幂等（run 级）保证不重复扣费 |

**边界（明确不做）**：不做 CRDT / OT 实时协同、不做离线自动三方合并、不承诺全功能离线。承诺范围严格为 L1~L3。
红线验收：断网可浏览+搜索+编辑并排队 AI 任务；恢复网络后草稿自动提交、出站 AI 任务自动重放且不重复扣费；并发编辑冲突必弹三方对比 UI，无静默覆盖。

## 10. 协作与权限（1~5 人）

users + project_members(project_id, user_id, role: owner/editor/viewer)。Owner 全权；Editor 读写内容/跑工作流，不可删项目/管成员/改预算；Viewer 只读。API 层装饰器校验 + 操作日志（audit_logs：谁/何时/对什么/做了什么）。**不做**：字段级权限、审批流、组织架构。修改记录由 C5 版本系统天然提供。

## 11. 安全与运维评审

清单项全部可低成本满足：JWT（短期 access + refresh）、API Key/平台账号 Fernet 加密（V1 持久化修复保留）、CSRF（SameSite + token）、XSS（Tiptap schema 白名单 + 输出转义）、SQL 注入（ORM 参数化 + CI 裸 SQL 检查）、审计日志（中间件级）、健康检查（/healthz + compose healthcheck）、每日备份（pg_dump 加密 → 异地，RPO≤24h/RTO≤1h，每月恢复演练）、告警（Telegram Bot）。

## 13. ⚠️ 遗留项修订与关闭（V2.1 核心变更）

本节逐条记录上一版 5 个 ⚠️ 项的修订内容与关闭依据，作为评审留痕。

### 13.1 多 Agent（⚠️ → ✅）
- **原问题**：10 个 Agent 若做成自治对话系统不可调试。
- **定案措施**：AgentSpec 契约固化 + `agent_registry` 统一实例化 + CI 静态守卫禁止 `agents/**` 相互 import + 工具白名单默认不含"调用其它 Agent" + Producer 只产出 WorkflowPlan 数据 + 每 Agent ≥1 golden case。（详见 §4.2）
- **关闭条件**：CI 守卫生效、golden case 全绿 → 关闭。

### 13.2 自动发布（⚠️ → ✅）
- **原问题**：国内平台自动化发布违反 ToS，有封号风险。
- **定案措施**：手动 / 半自动（默认）/ 全自动三模式；国内全自动默认关闭 + 二次确认 + 首篇强制 human 节点 + 连续失败自动降级 + 全程 audit_logs。（详见 §8.1）
- **关闭条件**：未开启时路径只能停在半自动待点击态；开启后首篇仍出现 human 确认节点 → 关闭。

### 13.3 Offline First（⚠️ → ✅）
- **原问题**："全功能离线"会拖入本地库同步冲突深水区。
- **定案措施**：严格限定 L1 只读 / L2 离线编辑（乐观锁 + 三方对比 UI）/ L3 AI 出站队列（幂等重放）；明确不做 CRDT / 自动合并 / 实时协同。（详见 §9）
- **关闭条件**：并发冲突必弹三方对比、无静默覆盖、出站任务不重复扣费 → 关闭。

### 13.4 风格学习 / 仿写（⚠️ → ✅）
- **原问题**：仿写可能过拟合到侵权级模仿。
- **定案措施**：入库闸（授权标注 + 第三方样本只抽统计特征）/ 生成闸（相似度阈值 0.75 强制重写、0.6~0.75 人工确认）/ 产物闸（发布网关拦截未过检产物）。（详见 §5.1）
- **关闭条件**：高相似输入被拦截并重写至阈值以下 → 关闭。

### 13.5 工程基线（有条件 → ✅ 门禁化）
- **原问题**：需先收口 V1 至 80 分再动 C1 迁移，否则在不稳定基座上重构。
- **定案措施**：V1 收口（TASK-001/002）列为 M1 首两任务，并**设为 C1 数据迁移（TASK-004）的硬前置门禁**——106 测试全绿 + slowapi 定型 + Celery retry/死信 + 备份告警实收后，方可执行迁移；迁移脚本自带前置备份与可回滚。（对应任务清单 M1）
- **关闭条件**：迁移 PR 的 CI 中，V1 收口门禁项全部为绿方允许合入 → 关闭。
