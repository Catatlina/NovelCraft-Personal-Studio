## 1. 通用规范

### 1.1 命名

- 文件/目录：`snake_case.py`（后端）、`kebab-case.tsx`（前端组件）、`camelCase.ts`（前端非组件）。
- 变量/函数：`snake_case`（Python）/ `camelCase`（TS）。
- 类/类型：`PascalCase`（两端一致）。
- 常量：`UPPER_SNAKE_CASE`。
- 禁止拼音缩写、禁止 `tmp`/`data2` 之类无意义名；命名要表达意图而非类型。

### 1.2 目录职责边界（引用技术方案 §1/§2）

后端（`backend/app/`）：
- `api/v1/`：仅做**请求解析 + 鉴权 + 调 service**，禁止写业务/DB 直查/LLM 调用。
- `services/`：业务逻辑；调 `repositories` 与 `ai.gateway`；C1~C8 能力在此实现。
- `repositories/`：数据访问；`ProjectScopedRepository` 强制归属过滤；`VersionedRepository` 封装版本。
- `ai/`：AI Gateway（gateway/router/budget/providers），LLM 唯一出口。
- `agents/`：AgentNode 定义（只读自己目录）。
- `models/`：SQLAlchemy ORM；`schemas/`：Pydantic；`core/`：跨层公共。

前端（`frontend/src/`）：
- `design/`：设计令牌/原子组件，禁业务耦合。
- `components/`：展示与交互，按域分目录。
- `pages/`：路由页面，组合组件。
- `stores/`：仅客户端态（zustand）；`api/`：生成客户端（禁手改）；`hooks/`：复用逻辑。
- `offline/`：SW/IndexedDB/outbox。

### 1.3 注释与文档字符串

- 公共函数/类必须有 docstring（Google 风格），说明「做什么、参数、返回、副作用」。
- 注释解释 **why** 而非 **what**；明显代码不加注释。
- 复杂算法（如 7 层装配、相似度检测）必须解释取舍与边界。
- 禁止遗留 `TODO` 无 issue 号；`TODO(TASK-012): ...`。

### 1.4 错误处理

- 不吞异常；底层异常向上抛，由 service/api 统一转换。
- 不向用户泄露堆栈/内部路径；统一错误响应格式（见 API 规范）。
- 外部调用（provider/平台 API）必须超时 + 重试 + 降级，不允许无限等待。

### 1.5 日志（structlog）

- 全程 `structlog`，禁止 `print` 调试（CI 可 grep 拦 `print(` 在 app 内）。
- 日志带结构化字段：`event` / `run_node_id` / `provider` / `cost` / `latency_ms`。
- 级别：DEBUG 调试、INFO 正常链路、WARNING 可恢复、ERROR 需告警。
- 禁止把密钥/明文内容打进日志；内容正文只记长度或摘要。

### 1.6 禁用项（红线）

- 禁止裸 SQL 拼接（防注入，CI 查 `execute(` 字符串格式化）。
- 禁止前端裸色值/魔法间距（stylelint 守卫）。
- 禁止 `any`（TS）/`# type: ignore` 无理由（Python）。
- 禁止 Agent 互调、LLM 绕过 Gateway、写操作绕过 VersionedRepository、内容直读绕过 content_service。
- 禁止提交空壳文档（见 §7）。

## 2. 后端 Python 规范

### 2.1 工具与严格度

- **ruff**：lint + format，零 warning 合入。
- **mypy `--strict`**：类型注解全覆盖，公共接口无 `Any`。
- 配置集中在 `pyproject.toml`：

```toml
[tool.ruff]
line-length = 100
target-version = "py311"

[tool.ruff.lint]
select = ["E","F","I","B","C4","UP","SIM","ASYNC"]
ignore = []

[tool.mypy]
strict = true
disallow_any_generics = true
warn_return_any = true
```

### 2.2 类型注解全覆盖

```python
from typing import Optional
from app.schemas.content import ContentOut

def get_chapter(project_id: int, chapter_id: int) -> Optional[ContentOut]:
    ...
```

- 函数签名、返回值、关键局部变量都要注解。
- 返回空用 `None` 而非隐式；集合用具体泛型 `list[ContentOut]`。

### 2.3 Pydantic Schema

- 所有 API 出入参走 Pydantic；`api` 层只接 `Schema` 不接裸 dict。
- content 的 `meta` 必须过 **meta Schema 注册表**校验（C1 红线，见《02》§2）：

```python
from pydantic import BaseModel, Field

class ChapterMeta(BaseModel):
    seq: int = Field(..., ge=1)
    volume_id: Optional[int] = None
    # 每个 type 的 meta 单独 Schema，注册到 CONTENT_META_REGISTRY

CONTENT_META_REGISTRY = {
    "chapter": ChapterMeta,
    "wechat_article": WechatArticleMeta,
    # 新增 type 必须在这里登记，CI 检查一致性
}
```

### 2.4 分层职责与调用方向

```
api/v1  ──► services  ──► repositories ──► DB
  │            │  ▲            │
  │            ▼  │            ▼
  │         ai.gateway ──► providers
  │            │
  └──── response (Schema)
```

铁律（CI grep 守卫）：
1. **LLM 只走 gateway**：任何 `openai`/`anthropic`/`requests` 直连 LLM 在 `app/` 内 → CI 拦截。
2. **写操作只走 VersionedRepository**：实体写路径必须 `VersionedRepository.save()`，禁止直接 ORM `session.add` 提交实体（读取可走 base repo）。
3. **内容读写只走 content_service**：`contents` 表读写必须经 `content_service`（含 meta 校验），禁止 service 直接 `ContentRepository` 裸写正文。

CI 守卫示例（伪）：
```bash
# LLM 直连守卫
grep -rEn "openai|anthropic|generativelanguage" backend/app --include=*.py \
  | grep -v "ai/providers" | grep -v "test" && exit 1
```

### 2.5 三条铁律（必背）

| # | 铁律 | 守卫 | 违规后果 |
|---|---|---|---|
| 1 | LLM 调用只准走 gateway | CI grep 禁止 `app/` 内直连 provider | 流水线失败 |
| 2 | 写操作只准走 VersionedRepository | 审计基类，直 `session.commit` 实体报 warning | 评审打回 |
| 3 | 内容读写只准走 content_service（含 meta 校验） | content_service 入口统一收口 | 评审打回 |

> 详见《05》§1 末尾铁律段与《07》§7 CI 段。

### 2.6 异步规范

- API 路由用 `async def`；DB 用 `asyncpg`/SQLAlchemy async。
- Celery 任务是**同步**函数（不要 `async def` task）；内部调用异步 gateway 用 `asyncio.run` 或事件循环包装。
- 不要在异步上下文用阻塞 IO；provider 调用走 aiohttp/httpx async 客户端。
- 信号量/锁：限流在 Redis，避免进程内锁跨 worker 失效。

### 2.7 异常层次

```python
class NovelCraftError(Exception): ...
class NotFoundError(NovelCraftError): ...
class PermissionError(NovelCraftError): ...   # 注意不覆盖内置，用 NCPermissionError 更佳
class BudgetExceededError(NovelCraftError): ...
class ProviderDegradedError(NovelCraftError): ...
class SchemaValidationError(NovelCraftError): ...
```

- 业务异常继承 `NovelCraftError`，由 api 中间件映射 HTTP 状态。
- 不抛裸 `Exception`；不捕获后静默。

### 2.8 Celery 任务幂等

- 任务必须**幂等**：同输入重跑不产生脏数据（C2 验收红线）。
- 用 `run_node_id` + `attempt` 做幂等键；重复投递先查状态。
- `acks_late=True` + `retry` 由 beat/worker 控制；重试指数退避。
- 失败节点单独重试不重跑全链（DAG 断点续跑）。
- 禁止在 task 内做不可重入的副作用（如重复发布）——发布走 adapter 幂等键。

## 3. 前端 TS / React 规范

### 3.1 组件结构

- 组件单文件单组件，默认导出组件、命名导出子部件。
- Props 用 `interface` 显式声明，禁 `any`/`object`。
- 展示组件不嵌业务请求，数据来自 props 或 store/hook。

```tsx
interface ChapterCardProps {
  chapterId: number;
  onOpen: (id: number) => void;
}

export function ChapterCard({ chapterId, onOpen }: ChapterCardProps) {
  return <button onClick={() => onOpen(chapterId)}>...</button>;
}
```

### 3.2 状态边界：zustand vs TanStack Query

- **zustand**：仅客户端态（auth、UI 选择、run 的本地 SSE 态、editor 草稿、offline 标志）。
- **TanStack Query**：服务端态（项目/内容/知识列表），离线时读 IndexedDB 持久化缓存。
- 禁止把服务端数据塞进 zustand 当缓存；禁止在组件里直接 `fetch` 绕过 Query。
- 写操作走 mutation，成功后 `invalidate` 相关 query。

```tsx
const { data } = useQuery({ queryKey: ["contents", projectId], queryFn: () => api.contents.list(projectId) });
const qc = useQueryClient();
const mutate = useMutation({ mutationFn: api.contents.create, onSuccess: () => qc.invalidateQueries({ queryKey: ["contents"] }) });
```

### 3.3 hooks 规范

- 自定义 hook 以 `use` 开头，返回稳定引用（用 `useCallback`/`useMemo`）。
- SSE：`useSSE` 统一订阅 `/api/v1/events`，推到 zustand run store。
- 离线：`useOffline` / `useAutosave` / `useBudget` / `useCommandK` 各司其职，逻辑复用。
- 不在 hook 内写业务分支大杂烩；复杂逻辑下沉 service 层（前端 `api/`）。

### 3.4 只用 design tokens（禁裸色值）

- 所有颜色/间距/圆角/阴影走 `design/tokens.css` 变量。
- 禁止 `#fff`、`24px` 之类裸值（stylelint 守卫拦截）。
- 暗色模式由 token 双套自动切换，组件不写 `prefers-color-scheme` 分支。

```css
/* 正确 */
.card { background: var(--color-surface); padding: var(--space-4); border-radius: var(--radius-md); }
/* 错误：裸色值，CI 拦截 */
.card { background: #ffffff; padding: 24px; }
```

### 3.5 Tiptap JSON 统一内容格式

- 正文统一 `Tiptap JSON` = `contents.body`（JSONB）；派生输出按平台序列化（MD/HTML/纯文本）。
- 禁止在数据库存 Markdown 原文当正文；跨平台格式转换在 content_service/发布 adapter 做。
- 编辑器扩展（AI 浮条/行内批注/版本树/diff）只消费 Tiptap JSON。

### 3.6 SSE / 离线 hooks

- `useSSE`：断线自动重连（指数退避），重连后补拉遗漏的 run_node 状态。
- `useAutosave`：编辑 1s 防抖 → 走 auto_save 版本（7 天清理）。
- `useOffline` + `outbox`：离线创建 AI 任务进 IndexedDB 出站队列，恢复网络后按序重放，依赖服务端 run 级幂等不重复扣费（L3）。
- L2 冲突：提交时若服务端 `base_version_id` 已变 → 生成分支进版本树 + 三方对比 UI，绝不静默覆盖。

### 3.7 a11y（可访问性）

- 语义化标签 + 表单 label 关联；交互元素可 Tab 聚焦。
- ⌘K 全局命令面板动作注册；快捷键不冲突。
- 颜色对比度达标（用 token 已保证）；图标按钮带 `aria-label`。
- 空态/加载态/错误态三态齐全（UI 验收五条之一，见《07》§5）。

## 4. Git 规范

### 4.1 分支策略

```
main      生产，只收 release/hotfix，合入即打 tag
develop   集成，日常目标分支
feat/task-0xx-简述   功能分支（绑定 TASK 编号）
fix/xxx   缺陷    hotfix/xxx  生产急修（main 切出，回灌 develop）
```

- 涉及**迁移 / Gateway / 工作流引擎 / 发布 / 版本系统**的改动必须走分支 + 自审 PR。
- 小改可直提 develop。

### 4.2 Commit 规范（中文详述）

格式：
```
<type>(<scope>): <中文简述>
- 变更点…
影响: API/DB/部署（无则写"无"）
测试: 新增/回归情况
```

- type：`feat`/`fix`/`refactor`/`security`/`perf`/`docs`/`test`/`chore`
- scope：`content`/`workflow`/`gateway`/`narrative`/`knowledge`/`publish`/`editor`/`design`/`infra`…
- **一 commit 一事**：不混功能与重构、不混迁移与业务。
- **迁移独立 commit**：Alembic 迁移文件单独一个 commit，便于回滚与 review。

示例：
```
feat(content): 新增 volume 类型章节聚合视图
- ContentService.list_volume_tree 按 parent_id 递归
- 增加 VolumeMeta schema 并登记注册表
影响: API(/contents 新增 query)、DB(无新表)
测试: 新增单测 test_volume_tree、回归 test_chapter_crud 通过
```

### 4.3 迁移纪律

- Alembic 线性单头、一 PR 一迁移、生产迁移自动前置备份、可降级。
- 迁移不可破坏旧数据；`downgrade` 必须有意义。
- 向量维度/embedding 变更属破坏性，需显式迁移并文档说明（换模型需迁移，明示）。

## 5. PR 与自审

### 5.1 改前《影响范围分析》

- 全仓调用链分析：改的函数被谁调用、会影响哪些 API/数据/前端/部署。
- 最小变更、向后兼容；安全检查（注入/XSS/密钥/越权）。
- AI 改动附加：重试幂等 / Prompt 版本化 / 上下文与 token 控制。
- 输出到 PR 描述首段。

### 5.2 改后《完成报告》

- 内容、测试结果（lint/type/unit/integration/build）、风险与建议、回滚方案。
- 关联 TASK 编号、影响分析、测试输出。

### 5.3 三轮自审

1. **开发者视角**：功能正确、边界处理、日志。
2. **架构师视角**：分层合规（三条铁律）、目录边界、扩展性、是否引入新基础设施（红线：不引入）。
3. **测试视角**：核心链路真实 pg+redis+TestClient 端到端，禁 mock DB/队列；失败路径覆盖；golden case 绿。

### 5.4 人工确认清单（必须你确认方可合入）

- [ ] Alembic 迁移（可降级、已评审）
- [ ] model_routes / 预算配置变更
- [ ] 发布 adapter（尤其国内平台自动化开关）
- [ ] 版本系统写路径（是否经 VersionedRepository）
- [ ] 依赖升级（security advisory 核对）
- [ ] 敏感词表变更

> 红线：未人工确认上述任一项，不得合入 main。

## 7. 文档纪律

- 「完成」的定义 = **代码合入 + 测试通过 + 文档可复现**。
- 补哪个模块的代码，同 PR 补该模块文档；文档 PR 与代码 PR 同评审。
- **禁止再出现只有标题的空壳文档**（本套 7 份为 V2 基线，原 37 份空壳按进度补实）。
- 文档改动随代码评审；API 变更必须同步《10-API 接口规范》；DB 变更同步《09-数据库设计文档》。
- 架构级变更必须新增 ADR（见《18-架构决策记录 ADR》）。
- 文档中示例配置/代码用代码块 + ASCII 直引号，与代码仓库一致。

## 附录 A：测试约定

- 测试目录 `backend/app/tests/`，按层分 `api/` `services/` `repositories/` `ai/`。
- 核心链路必须真实 Postgres + Redis + TestClient 端到端，**禁 mock 数据库与队列**（《07》§3 测试铁律）。
- 每个 AgentNode ≥1 条 golden case：固定 input → 断言 output schema + 关键字段，进 CI。
- 改 Prompt 必跑 golden case；改 model_routes 必带降级链测试（主失败→备→PENDING_PROVIDER）。
- 前端：组件单测（Vitest + Testing Library）+ 关键 hook 测试；e2e 至少覆盖「注册→建项目→跑工作流→看产出」。

## 附录 B：反模式清单（踩坑即打回）

| 反模式 | 正确做法 |
|---|---|
| api 层直接查 DB / 调 LLM | 只解析请求 → 调 service；LLM 走 gateway |
| service 直接 `session.add(entity)` 提交 | 走 `VersionedRepository.save()` |
| 直接 `ContentRepository` 裸写 `contents.body` | 经 `content_service`（带 meta 校验） |
| `agents/` 下 import 另一个 `agents/` | 只经 `agent_registry` + DAG 边推进 |
| 前端 `fetch` 绕过 TanStack Query | 用 Query/mutation + invalidate |
| 组件写 `#fff` / `24px` | 用 `var(--color-*)` / `var(--space-*)` |
| 数据库存 Markdown 当正文 | 统一 Tiptap JSON，派生时再序列化 |
| 裸 `print()` 调试 | `structlog` 结构化日志 |
| 一个 commit 混功能+重构+迁移 | 拆 commit，迁移独立 |
| 空壳文档只留标题 | 同 PR 补实文档 |
