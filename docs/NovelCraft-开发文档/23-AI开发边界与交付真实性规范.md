# NovelCraft AI 开发边界与交付真实性规范

> 版本：V1.1 ｜ 状态：强制执行 / 每次开发前必读 ｜ 适用对象：所有 AI Agent、自动编码工具和人工开发者
>
> 目标：禁止“文件存在即完成、函数可导入即完成、文档改勾即完成”，确保每次进度上报可以被代码、测试和用户入口复核。

## 0. 开发前必读门禁（每一轮必须执行）

每次开始开发、整改、审计修复、提交前，执行者必须先阅读本规范，并在计划或交付说明中显式遵守以下门禁：

1. **先定状态口径**：任何能力只能使用 `未开始 / 已接线 / 可用 / 已验收` 四档对外描述；不得把“骨架、文件存在、函数可 import、路由存在”说成完成。
2. **先看当前进度**：开发前必须阅读 `PROJECT_PROGRESS.md`、当前执行队列（如 `37-新增需求任务分解-20260713.md`）和最新审计报告；不得引用已归档旧报告作为完成率依据。
3. **先查假完成风险**：开发前和交付前都必须运行或等价执行 §10 的反撒谎扫描；命中项必须整改或说明为何不影响产品路径。
4. **先明确外部依赖**：依赖真实 AI Key、平台账号、API、cookie、TLS、外部数据源的功能，未实测前只能写“配置已具备 / 未真实验收”，不得写完成。
5. **先保护用户改动**：工作树存在用户或其他 Agent 改动时，不得 reset、checkout、覆盖或顺手重构无关文件。
6. **先定验收命令**：每个任务开始前要明确对应后端测试、前端构建、E2E、真实 Provider smoke 或外部平台验收命令；做不到就不能把目标定义为“完成”。

违反本节任一项，交付结论必须降级；已经上报的完成状态必须在同一轮更正。

## 0.1 对外状态四档（强制口径）

| 状态 | 允许含义 | 禁止话术 |
|---|---|---|
| 未开始 | 没有代码或仅有需求描述 | 不得计入完成率 |
| 已接线 | 有前端/接口/服务/表/任务中的部分链路，但缺完整验收 | 不得说“可用”“完成”“融合完成” |
| 可用 | 用户能通过正式入口跑通主路径；有本地测试或 E2E 证据 | 不得覆盖真实平台未验收边界 |
| 已验收 | 有 commit、命令、输出、时间、文件、测试等级证据；外部依赖已实测 | 不得用 monkeypatch 或 T0 存在性测试替代 |

内部可继续用 `🧱 骨架 / 🚧 部分完成 / 🧪 待验收 / ✅ 已完成 / ⛔ 阻塞`，但对用户、报告、进度文档必须映射到上述四档，避免“部分完成”被误读为完成。

## 0.2 完成声明的六项必要条件

任何功能宣称“可用”或“已验收”，必须同时满足：

1. 有正式前端入口；纯后端能力必须明确标注“纯后端”。
2. 有正式后端 API / service / worker 调用链，不是孤立函数。
3. 有数据库、配置、权限、项目归属、幂等或状态机闭环。
4. AI 能力必须真实调用 `app.gateway.complete()` / `complete_stream()` 或明确的真实 Provider 适配，并写入 `ai_calls`；禁止 mock、fallback、固定模板、启发式冒充 AI。
5. 失败必须显式报错或进入 `failed`，不得静默返回空列表、degraded 成功、默认草稿冒充成功。
6. 有成功、失败、权限/隔离、持久化测试；用户主链必须有 E2E；真实平台能力必须有真实凭据/回执证据。

缺任一项，最高只能写“已接线”或“配置已具备 / 未真实验收”。

## 1. 权限与范围边界

1. AI 只能修改用户已授权的仓库、分支和功能范围；不得自行发布生产、申请外部账号、购买服务或扩大许可证义务。
2. 用户要求“审查/报告”时只读；要求“开发/修复”时才允许改代码；要求“提交/实时上传”时才允许提交推送。
3. 不得删除或覆盖无法确认归属的用户修改；不得用 reset/checkout 丢弃工作区内容。
4. GPL/AGPL 默认禁止复制源代码、Prompt 和实现结构；能力通过公开行为规格独立实现。MIT/Apache 复用必须保留归属、许可证和修改记录。
5. 抓榜、拆书和仿写只处理合法公开元数据或用户有权使用的文本；不得把参考作品可识别表达送入成书 Prompt。

## 2. 唯一状态定义

| 状态 | 含义 | 必要证据 |
|---|---|---|
| ⬜ 未开始 | 没有可执行实现 | 可以有设计文档，不得计算为完成 |
| 🧱 骨架 | 有表、接口签名、组件或函数，但未形成用户结果 | 文件/符号定位 |
| 🚧 部分完成 | 主路径的一部分可运行，仍缺入口、持久化、错误处理或关键节点 | 已完成与未完成边界 |
| 🧪 待验收 | 功能链已实现，尚未通过真实 E2E/外部环境验证 | 自动测试结果与待验收条件 |
| ✅ 已完成 | 用户可从正式入口完成目标，数据可追溯，失败可恢复，验收门禁通过 | commit + 实现定位 + 测试编号 + 验收输出 |
| ⛔ 阻塞 | 需要用户授权、凭据或外部状态 | 阻塞证据和解除条件 |

“有代码但未实测”必须标记为 `🧪`，不能为了消灭模糊状态改成 `✅`。

## 3. `✅`的六项硬门禁

1. **实现**：不是空函数、固定字典、mock、占位返回或仅复制上游材料。
2. **入口**：后端有正式 API/任务入口；用户功能有可发现的前端入口。
3. **持久化**：业务结果进入正式数据模型，并保留项目归属与来源血缘。
4. **失败语义**：超时、外部源失败、校验失败和重试耗尽不能伪装成成功或空列表。
5. **测试**：至少覆盖成功、失败、权限/隔离、幂等；核心流程还要有 E2E。
6. **验收**：测试实际执行通过，记录 commit、命令、日期和未覆盖边界。

任一门禁缺失，最高只能标为 `🚧` 或 `🧪`。

## 4. 禁止的虚假上报

- 仅凭文件、类、函数、路由或组件存在标记完成。
- 测试只断言 `callable`、`isinstance`、路由存在，却宣称业务完成。
- 外部采集失败返回空值或 degraded，测试仍通过并宣称真实数据源完成。
- 用固定模板/固定字典冒充 AI 分析、分层规划、合理性审计或 humanize。
- 复制上游仓库、Skill、Prompt 或创建 gitlink 后宣称完成融合。
- 说明写“未实现、待实现、未实测、部分、骨架、未接”，状态却写 `✅`。
- 多个子能力共用一个存在性测试，却按子能力数量重复统计。
- 修改数字、表头或状态符号提高完成率，却没有对应代码和测试证据。
- 隐瞒失败测试、环境限制、外部凭据、安全或许可证风险。

## 5. 测试证据等级

| 等级 | 证据 | 能证明什么 |
|---|---|---|
| T0 | import/文件/函数存在 | 仅证明骨架 |
| T1 | 纯函数单测 | 局部逻辑正确 |
| T2 | API + 数据库集成测试 | 接口、权限、持久化正确 |
| T3 | Worker/Redis/Provider 故障测试 | 异步、重试、恢复正确 |
| T4 | 浏览器 E2E | 用户从前端可完成目标 |
| T5 | 真实外部源/长时间运行 | 真实可用性与稳定性 |

核心主链“扫榜→分析→选题→整书→书库”必须达到 T4；榜单源、自动连载和发布适配器还必须达到 T5 才能标记完全交付。

## 6. 每次交付报告

必须列出：Commit/PR、用户可见结果、实现入口、数据迁移、测试命令与结果、证据等级、尚未覆盖、外部阻碍、同步文档。禁止只报告完成项数量。

## 7. 开源融合完成定义

每个上游能力必须在 `upstream_manifest` 记录仓库、commit、许可证、行为规格、实现文件、API/节点、数据模型、测试编号、差异和未实现项。完成融合还必须满足：

1. 已转换为 NovelCraft 数据协议，不依赖上游应用运行；
2. 通过 NovelCraft Gateway、工作流和权限体系；
3. 有用户入口和持久化；
4. 有行为对照与失败测试；
5. 许可证归属和独立实现证据完整。

严禁以下融合虚报：

- 把 `upstream/` 中存在的源码、README、Skill、Prompt 当作融合完成。
- 把 deprecated 模块、无活跃调用方模块、孤立 helper 函数计入 active/integrated。
- 把 `available: true`、`wired: true`、`status: active` 写成硬编码自报告；状态必须由真实入口、调用方、测试和运行证据驱动。
- 把固定模板、正则启发式、字符串长度评分、示例数据包装成上游能力融合。
- 把“设计稿”“规划函数”“未接执行器 DAG”“人工草稿导出”说成自动执行或全自动发布。

开源项目只有在“进入真实产品路径 + 真实 AI/数据/权限/持久化闭环 + 行为测试通过”后，才能写“融合完成”。否则必须写“参考吸收 / 已接线 / 部分可用 / 已废弃 / 未验收”。

## 8. 进度治理

- `PROJECT_PROGRESS.md` 是当前唯一权威进度摘要。
- 《22-全功能跟踪表》旧版统计失真，已废止；重建证据台账前不得引用其完成数。
- 每个状态变化必须与同一提交中的实现或验收证据绑定。
- CI 执行 `scripts/verify_delivery_claims.py`，发现完成状态与否定说明冲突时失败。
- 发现虚假上报立即降级状态并记录原因，不允许用新百分比覆盖历史问题。

## 9. 功能优先顺序

1. 扫榜成书、统一书库；
2. 热点自媒体；
3. 长篇一致性与六项目真实融合；
4. 发布数据回流；
5. 出海、协作和增强能力。

不得为了提高完成项数量，优先实现大量与主链无关的简单函数。

## 10. 反撒谎扫描与交付前检查

每次开发前、交付前、提交前，必须执行以下扫描或等价审计；命中项必须逐条处理，不能用“测试通过”掩盖。

```bash
rg -n "mock|fake|fallback|placeholder|TODO|FIXME|DEPRECATED|No active callers|NotImplemented|except: pass|return \\{\\}|return \\[\\]" backend/app frontend/src
rg -n "震惊！|你不知道的|固定模板|Estimated|hardcoded|demo" backend/app frontend/src
rg -n "deep_.*wired.*True|available.*True|status.*active" backend/app/services backend/app/api
rg -n "provider.*degraded|degraded.*200|status.*succeeded" backend/app
git diff --check
```

扫描解释规则：

1. 测试代码可使用 monkeypatch，但不得用 monkeypatch 证明 AI 能力完成。
2. `DEPRECATED` 或 `No active callers` 模块不得出现在 active 融合状态或产品完成率中。
3. `fallback` 只允许出现在“拒绝 fallback / 迁移历史字段 / 文档解释”语境；不得出现在业务生成路径。
4. `return []` / `return {}` 如是合法空态必须有错误语义、空态文案或调用方处理；不得冒充成功。
5. 任何 `available: true` / `wired: true` / `status: active` 必须能从实际路由、调用方和测试证据推导。

## 11. 固定开发顺序

后续开发必须按以下顺序执行，禁止跳级：

1. 修复 P0 阻断：白屏、崩溃、伪数据、伪 AI、越权、密钥泄露、迁移失败。
2. 修复 P1 上线前必需：真实调用闭环、发布/采集凭据闭环、状态机、错误语义、融合自报告真实性。
3. 补 P2 质量与体验：文档对齐、提示优化、前端单测、E2E 扩展、长期稳定性。
4. 每完成一项立即补测试；测试未补不得标可用。
5. 每轮最后更新 `PROJECT_PROGRESS.md` 和对应开发文档，只写已验证事实。
6. 提交前跑相关测试、构建、E2E、密钥扫描；未跑必须写明“未执行”和原因。

## 12. 标准交付汇报格式

每次交付必须使用以下结构，禁止只写“已完成/已修复/全部通过”：

```text
已完成：
- 功能：
- 文件：
- 验证命令：
- 验证结果：
- 证据等级：

未完成：
- 项目：
- 原因：
- 阻断条件：
- 下一步：

不能宣称完成的项：
- 项目：
- 原因：
- 当前只能标记为：
```

凡依赖真实平台账号/API、真实 AI Key、生产 TLS、真实发布回执、长周期真实采集的能力，如没有实测证据，必须写“配置已具备 / 未真实验收”，不得写“完成”。

## 13. AI_EXECUTION_CONTRACT（机器可读执行契约）

以下 YAML 块是给所有 AI Agent / 自动化工具读取的最小执行契约。若自然语言文档与本块冲突，以更严格者为准。

```yaml
AI_EXECUTION_CONTRACT:
  version: "1.1"
  mandatory: true
  must_read_before_every_task:
    - "AGENTS.md"
    - "PROJECT_PROGRESS.md"
    - "docs/NovelCraft-开发文档/23-AI开发边界与交付真实性规范.md"
    - "docs/NovelCraft-开发文档/37-新增需求任务分解-20260713.md"
    - "latest_user_provided_audit_report_if_any"

  mandatory_gate_command:
    - "bash scripts/ai_development_gate.sh"

  allowed_public_status:
    - "未开始"
    - "已接线"
    - "可用"
    - "已验收"

  status_rules:
    未开始:
      requires: ["no_executable_implementation"]
      cannot_claim: ["completion_percentage"]
    已接线:
      requires: ["some_code_or_entry_exists"]
      cannot_claim: ["可用", "完成", "已验收", "融合完成"]
    可用:
      requires:
        - "formal_user_or_backend_entry"
        - "real_runtime_path"
        - "local_test_or_e2e_passed"
        - "explicit_failure_semantics"
      cannot_claim: ["external_platform_verified_without_real_receipt"]
    已验收:
      requires:
        - "commit_or_worktree_reference"
        - "commands_and_outputs"
        - "date"
        - "files_changed"
        - "evidence_level_T2_or_higher"
        - "external_dependencies_verified_if_in_scope"

  completion_required_conditions:
    - "frontend_entry_or_explicit_backend_only_boundary"
    - "backend_api_service_worker_call_chain"
    - "database_config_permission_project_scope_closure"
    - "real_ai_provider_path_for_ai_features"
    - "ai_calls_provenance_for_ai_features"
    - "explicit_error_or_failed_state_on_failure"
    - "success_failure_permission_persistence_tests"
    - "e2e_for_user_main_chain"

  prohibited_patterns:
    product_code:
      - "mock result"
      - "fallback generated result"
      - "static template presented as AI"
      - "heuristic presented as AI"
      - "degraded result wrapped as success"
      - "silent empty success"
      - "deprecated module reported active"
      - "hard-coded wired:true / available:true"
      - "upstream files counted as fused"
      - "skeleton counted as complete"
    reporting:
      - "all complete without evidence"
      - "fully fused without active product path"
      - "real platform verified without credentials and receipt"
      - "tests passed but commands not shown"

  ai_feature_rules:
    must_use_one_of:
      - "app.gateway.complete"
      - "app.gateway.complete_stream"
      - "explicit real provider adapter with failure propagation"
    must_record:
      - "ai_calls row or explicit provenance record"
    forbidden:
      - "mock provider"
      - "fallback_json as output source"
      - "fixed dict/string output"
      - "regex/length score presented as AI judgement"
      - "provider failure converted to ok"

  fusion_rules:
    complete_requires:
      - "active product caller"
      - "formal API/UI/workflow entry"
      - "NovelCraft data protocol"
      - "permissions and project scope"
      - "persistence"
      - "behavior test"
      - "license boundary"
    forbidden_complete_sources:
      - "upstream directory only"
      - "README only"
      - "deprecated module"
      - "helper with no caller"
      - "design draft"
      - "manual-only draft export"

  external_dependency_rules:
    if_no_real_credentials_or_receipt:
      max_status: "已接线"
      required_phrase: "配置已具备 / 未真实验收"
    applies_to:
      - "AI provider keys"
      - "platform publishing accounts"
      - "hotspot cookies/tokens/proxies"
      - "TLS/domain/certificates"
      - "remote embedding providers"
      - "Sentry/Telegram alerts"

  test_rules:
    monkeypatch_allowed_for:
      - "protocol"
      - "rollback"
      - "permissions"
      - "failure path"
      - "network isolation"
    monkeypatch_not_allowed_to_prove:
      - "real AI capability completion"
      - "real platform completion"
      - "external source stability"
    required_for_completion:
      - "relevant backend tests"
      - "frontend build when UI changed"
      - "e2e when user flow changed"
      - "secret scan before commit"

  required_final_response_sections:
    - "已完成"
    - "未完成"
    - "不能宣称完成的项"
    - "验证命令与结果"
```
