# NovelCraft AI 开发边界与交付真实性规范

> 版本：V1.0 ｜ 状态：强制执行 ｜ 适用对象：所有 AI Agent、自动编码工具和人工开发者
>
> 目标：禁止“文件存在即完成、函数可导入即完成、文档改勾即完成”，确保每次进度上报可以被代码、测试和用户入口复核。

## 1. 权限与范围边界

1. AI 只能修改用户已授权的仓库、分支和功能范围；不得自行发布生产、申请外部账号、购买服务或扩大许可证义务。
2. 用户要求“审查/报告”时只读；要求“开发/修复”时才允许改代码；要求“提交/实时上传”时才允许提交推送。
3. 不得删除或覆盖无法确认归属的用户修改；不得用 reset/checkout 丢弃工作区内容。
4. GPL/AGPL 默认禁止复制源代码、Prompt 和实现结构；能力通过公开行为规格独立实现。MIT/Apache 复用必须保留归属、许可证和修改记录。
5. 抓榜、拆书和仿写只处理合法公开元数据或用户有权使用的文本；不得把参考作品可识别表达送入成书 Prompt。

## 2. 唯一状态定义

| 状态 | 含义 | 必要证据 |
|---|---|---|
| ⬜ 未开始 | 没有可执行实现 | 可以有设计文档，不得计算为完成 |
| 🧱 骨架 | 有表、接口签名、组件或函数，但未形成用户结果 | 文件/符号定位 |
| 🚧 部分完成 | 主路径的一部分可运行，仍缺入口、持久化、错误处理或关键节点 | 已完成与未完成边界 |
| 🧪 待验收 | 功能链已实现，尚未通过真实 E2E/外部环境验证 | 自动测试结果与待验收条件 |
| ✅ 已完成 | 用户可从正式入口完成目标，数据可追溯，失败可恢复，验收门禁通过 | commit + 实现定位 + 测试编号 + 验收输出 |
| ⛔ 阻塞 | 需要用户授权、凭据或外部状态 | 阻塞证据和解除条件 |

“有代码但未实测”必须标记为 `🧪`，不能为了消灭模糊状态改成 `✅`。

## 3. `✅`的六项硬门禁

1. **实现**：不是空函数、固定字典、mock、占位返回或仅复制上游材料。
2. **入口**：后端有正式 API/任务入口；用户功能有可发现的前端入口。
3. **持久化**：业务结果进入正式数据模型，并保留项目归属与来源血缘。
4. **失败语义**：超时、外部源失败、校验失败和重试耗尽不能伪装成成功或空列表。
5. **测试**：至少覆盖成功、失败、权限/隔离、幂等；核心流程还要有 E2E。
6. **验收**：测试实际执行通过，记录 commit、命令、日期和未覆盖边界。

任一门禁缺失，最高只能标为 `🚧` 或 `🧪`。

## 4. 禁止的虚假上报

- 仅凭文件、类、函数、路由或组件存在标记完成。
- 测试只断言 `callable`、`isinstance`、路由存在，却宣称业务完成。
- 外部采集失败返回空值或 degraded，测试仍通过并宣称真实数据源完成。
- 用固定模板/固定字典冒充 AI 分析、分层规划、合理性审计或 humanize。
- 复制上游仓库、Skill、Prompt 或创建 gitlink 后宣称完成融合。
- 说明写“未实现、待实现、未实测、部分、骨架、未接”，状态却写 `✅`。
- 多个子能力共用一个存在性测试，却按子能力数量重复统计。
- 修改数字、表头或状态符号提高完成率，却没有对应代码和测试证据。
- 隐瞒失败测试、环境限制、外部凭据、安全或许可证风险。

## 5. 测试证据等级

| 等级 | 证据 | 能证明什么 |
|---|---|---|
| T0 | import/文件/函数存在 | 仅证明骨架 |
| T1 | 纯函数单测 | 局部逻辑正确 |
| T2 | API + 数据库集成测试 | 接口、权限、持久化正确 |
| T3 | Worker/Redis/Provider 故障测试 | 异步、重试、恢复正确 |
| T4 | 浏览器 E2E | 用户从前端可完成目标 |
| T5 | 真实外部源/长时间运行 | 真实可用性与稳定性 |

核心主链“扫榜→分析→选题→整书→书库”必须达到 T4；榜单源、自动连载和发布适配器还必须达到 T5 才能标记完全交付。

## 6. 每次交付报告

必须列出：Commit/PR、用户可见结果、实现入口、数据迁移、测试命令与结果、证据等级、尚未覆盖、外部阻碍、同步文档。禁止只报告完成项数量。

## 7. 开源融合完成定义

每个上游能力必须在 `upstream_manifest` 记录仓库、commit、许可证、行为规格、实现文件、API/节点、数据模型、测试编号、差异和未实现项。完成融合还必须满足：

1. 已转换为 NovelCraft 数据协议，不依赖上游应用运行；
2. 通过 NovelCraft Gateway、工作流和权限体系；
3. 有用户入口和持久化；
4. 有行为对照与失败测试；
5. 许可证归属和独立实现证据完整。

## 8. 进度治理

## 9. 功能优先顺序

1. 扫榜成书、统一书库；
2. 热点自媒体；
3. 长篇一致性与六项目真实融合；
4. 发布数据回流；
5. 出海、协作和增强能力。

不得为了提高完成项数量，优先实现大量与主链无关的简单函数。
