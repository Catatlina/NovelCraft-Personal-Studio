# NovelCraft Personal Studio 工程协作规范（重制版）

> 适用：你 + Claude 协作开发，后续兼容 1~5 人小团队。延续既有工作协议并新增 UI/文档纪律。

## 1. Git 分支策略

```
main      生产，只收 release/hotfix，合入即打 tag
develop   集成，日常目标分支
feat/task-0xx-简述   功能分支（绑定 TASK 编号）
fix/xxx   缺陷    hotfix/xxx  生产急修（main 切出，回灌 develop）
```
小改可直提 develop；涉及**迁移 / Gateway / 工作流引擎 / 发布 / 版本系统**的改动必须走分支 + 自审 PR。

## 2. Commit 规范（中文详述）

```
<type>(<scope>): <中文简述>
- 变更点…
影响: API/DB/部署（无则写"无"）
测试: 新增/回归情况
```
type: feat/fix/refactor/security/perf/docs/test/chore；scope: content/workflow/gateway/narrative/knowledge/publish/editor/design/infra…；一 commit 一事；迁移文件独立 commit。

## 3. 代码修改流程（强制，延续既有协议）

**改前**《修改影响范围分析》：全仓调用链分析；影响面（文件/调用/数据/API/前端/部署）；最小变更、向后兼容；安全检查（注入/XSS/密钥/越权）；AI 改动附加检查（重试幂等/Prompt 版本化/上下文与 token 控制）。
**改后**《修改完成报告》：内容、测试结果（lint/type/unit/integration/build）、风险与建议。
**三轮自审**：开发者→架构师→测试视角。
**测试铁律**：核心链路真实 Postgres/Redis/TestClient 端到端，禁 mock 数据库与队列；全绿才打包部署。

## 4. Review 与人工确认清单

Claude 三轮自审 + 你抽查。以下**必须你人工确认**方可合入：Alembic 迁移、model_routes/预算配置、发布 adapter（尤其国内平台自动化）、版本系统写路径、依赖升级、敏感词表变更。PR 必附：TASK 编号、影响分析、测试输出、回滚方案。

## 5. UI 验收规范（UI 第一的制度化）

每个页面合入前过五条：① 只用 design tokens（禁裸色值/魔法间距，CI stylelint 守卫）；② 明暗双模式截图对照；③ 空态/加载态/错误态三态齐全；④ 键盘可达（Tab 顺序 + ⌘K 动作注册）；⑤ 与对标产品同类页面并排截图自评（Linear/Notion/Cursor 任一）。工作台/编辑器页由你亲自验收，管理页可 Claude 自审。

## 6. Issue 管理

GitHub Issues；标签：里程碑(M1~M5)、P0~P3、bug/feat/tech-debt、模块。一 TASK 一 Issue，验收标准写入正文逐项勾选后关闭；bug 必附复现与根因。

## 7. CI/CD

```
push/PR → ruff + mypy + tsc + stylelint(tokens守卫)
        + 单测(pg+redis 服务容器) + prompt golden case 回归
        + LLM 直连 grep 守卫 + docker build
merge main → tag → 构建镜像 → 手动触发 make deploy
           （备份→migrate→up→healthcheck→失败自动回滚上一镜像）
```

## 8. 测试与发版门禁

新功能必带测试（含失败路径）；TASK 验收标准即集成用例；改 Prompt 必跑 golden case；里程碑发版加：备份恢复演练 + 30 分钟手工探索测试 + 全功能映射表勾验（M5）+ 报告归档 docs/reports/。

## 9. 文档纪律

本套 7 份文档为 V2 唯一基线；原 37 份空壳文档按开发进度逐份补实（补哪个模块的代码，同 PR 补该模块文档），**文档 PR 与代码 PR 同评审**；"完成"的定义 = 代码合入 + 测试通过 + 文档可复现。禁止再出现只有标题的文档。
