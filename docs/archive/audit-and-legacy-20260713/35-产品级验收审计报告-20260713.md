## 0. 一句话结论

**这个项目现在不能直接交付给作者使用，无法完成"从零创作一本新小说"的核心闭环。**
后端不是空壳（代码量与工程治理都相当扎实），但**最新主打的"四阶段20节点"创作工作流是一个未接线的门面**——20 个节点在代码里定义了，却没有配任何模型路由、没有配任何提示词，默认模型指向一个**DeepSeek 真实 API 中并不存在的虚构模型 `deepseek-v4-pro`**。真实用户点击"生成小说"会在**第 1 个节点就失败**。

**上线评估：C —— 核心功能存在严重缺陷，不可上线。**

## 1. 项目架构分析报告

### 1.1 技术栈（属实）
| 层 | 技术 | 核实结论 |
|---|---|---|
| 后端 | FastAPI + psycopg2 + PostgreSQL(pgvector) | 真实，144 个 py 文件 |
| 任务队列 | Celery + Redis | 真实，`execute_bootstrap` 等任务真实存在 |
| 前端 | React 19 + TypeScript + Vite | 真实，29 个组件，约 3000 行 TS/TSX |
| AI | DeepSeek 主 + Claude/OpenAI/Gemini 降级链 | 调用代码真实（urllib/httpx），但**路由未接线**（见 §5） |
| 迁移 | Alembic（21 个版本） | 真实，但**存在多 head + 一处崩溃迁移**（见 Bug-02/03） |

### 1.2 架构真实性判断
- **AI 网关 `gateway.py`（789 行）**：真实实现。含预算门禁、断路器、失败 ledger（`pending_provider`）、输出契约校验、schema 重试。**并非假实现**。
- **Mock 守卫**：`provider=="mock"` 时强制要求 `NOVELCRAFT_ENV=test` 且 `ALLOW_MOCK=true`，否则抛错。**生产不会静默造假**——这一点值得肯定。
- **工作流 `workers/tasks.py`（82KB）**：四阶段 20 节点、幂等键、章节门禁、checkpoint 续跑、事件账本，逻辑真实。
- **文档与治理**：`PROJECT_PROGRESS.md` 自带 `delivery-claims:strict`（严格交付声明） 规则，并诚实记录了"默认 AI 路径为 mock""历史多轮虚假上报被降级"。**项目方对自身状态是有自知的。**

### 1.3 结构性风险
- 文档量 292 个 md vs 前端仅 34 个 ts/tsx —— **文档远重于可运行代码**，"完成度"叙事主要由文档支撑。
- `main.py` 单文件 57KB、`tasks.py` 82KB —— 巨型文件，维护性差。

## 3. 核心流程测试报告（真实环境实测）

### 3.1 测试环境搭建（真实执行）
```
docker run pgvector/pgvector:pg16           # 真实 Postgres
alembic upgrade 9f09aa0f10cc                 # 真实迁移（主分支）
python -c "from app.db import init_db; init_db()"   # 真实播种
```
均成功执行，37 张表建成，31 条 model_routes + 18 条 prompts 落库。

### 3.2 决定性验证：主创作流程首节点解析
真实调用 `gateway._load_prompt_and_route('bootstrap.plan_idea','plan_idea', ...)` 的输出：

```
=== 旗舰工作流首节点 plan_idea 实际解析 ===
provider : deepseek
model    : deepseek-v4-pro   <-- DeepSeek 真实 API 无此模型！
prompt   : '请执行任务 plan_idea，并输出 JSON。'   <-- 15字通用兜底，非任何精心设计的方法论提示词

=== 对比 旧节点 gen_titles（已播种）===
provider : deepseek | model: deepseek-chat   <-- 真实模型
prompt   : '你是资深网文编辑。请根据以下设定生成 5 个具有商业感…（完整方法论）'
```

### 3.3 数据库落库实测（决定性证据）
```
model_routes 中 18 个新节点 task_type 命中数：0   （应为 18）
prompts    中 18 个新节点提示词命中数：       0   （应为 ~18）
实际 routes 全部是旧 task_type（gen_titles/editor_*/…），全部 model=deepseek-chat
```

### 3.4 用户旅程推演（基于实测）
**旅程 A：作者创建新书 → 点击"生成小说"**
1. 前端 `POST /api/v1/novels/{id}/bootstrap` → `create_run()` → 播种 20 个 `BOOTSTRAP_NODES` → `execute_bootstrap.delay(run_id, "plan_idea")`（已核实调用链）
2. 首节点 `plan_idea` 执行 → `_load_route("plan_idea")` 返回 None → 模型回退到 `deepseek-v4-pro`
3. **情形①（作者未在设置里手填模型）**：DeepSeek 返回"模型不存在"→ `ProviderError` → 节点置 `pending_provider` → **工作流停在第 1 步，一个字都写不出来。**
4. **情形②（作者手填 model=deepseek-chat）**：API 能调通，但**每个节点都用那句 15 字通用提示词**，且新节点**无任何输出契约校验**（`validate_task_output` 对无 schema 的 task_type 直接放行）→ 产出未经工程化的散乱内容；下游 `_enrich_blueprint_context` 按 `kind` 读 `knowledge_items` 也难以拿到结构化数据 → **级联退化，写不成一本结构完整的书。**

**旅程 B：对已有小说"续写下一章"** → 走 `gen_next_chapter`（路由 deepseek-chat + 提示词齐全）→ **配好真实 key 时这条路是通的。**

> 结论：**"从零写一本新书"的闭环是断的；"在已有书上续写"这条支路是通的。** 而产品主打与最新提交主推的恰恰是前者。

## 4. "445 passed" 测试可信度评估

- 全套测试在 `conftest.py` 中强制 `NOVELCRAFT_ENV=test` + `ALLOW_MOCK=true`，**整体跑在 mock 之上，不触达真实 AI**。
- 触及 `execute_bootstrap` 的测试要么 **monkeypatch 掉 `_load_prompt_and_route` 直接返回 `("prompt","mock","mock",{})`**（`test_bootstrap_reliability.py`），要么只做**静态属性断言**（`test_ci.py`: `max_retries==3`），要么用 `inspect.getsource` **检查源码字符串**（`test_m2_final.py`）。
- **没有任何一个测试验证 `plan_idea` 等新节点能否解析到有效的路由与提示词。** 播种缺口恰好被 mock 与 monkeypatch 掩盖。

> 因此 "445 passed / 19/19 节点 succeeded" **不能证明真实创作闭环可用**，只证明了 mock 管路与被打桩的路径通畅。

## 6. 功能可用性报告（可用 / 假实现 / 缺失）

**真实可用（无需 AI 或旧路径已接线）**
- 用户注册登录、JWT、项目成员权限
- 书库增删改查、搜索、分页
- 续写下一章、编辑器润色/改写/去AI味、大纲扩展（均为旧 task_type，已接线；需真实 key）

**假实现 / 门面（有页面有代码，实际跑不通）**

**缺失 / 未闭环**
- "从零到一本完整小说"的端到端闭环
- 新节点的方法论提示词（StoryArchitect/Character/Reviewer 各阶段）与输出契约
- 真实 provider 的端到端回归测试（当前全 mock）

## 8. 项目整体评估报告 & 上线评估

### 8.1 综合判断
- **不是空壳**：后端引擎、治理机制、旧创作路径都是真材实料，工程品味在同类个人项目里偏上。
- **但主打功能是门面**：最新提交高调宣称的"四阶段20节点 + v4-pro模型"在真实库里 **0/18 接线**，默认模型是虚构的，**真实作者无法用它从零写完一本新书**。
- **另有两个阻断级部署 bug**：商业化迁移崩溃、迁移多 head。

### 8.2 上线评估（四选一）

> # 🅲 —— 核心功能存在严重缺陷，不可上线

**不选 A/B**：主创作闭环断在第一步，非"小修"可解（需补 18 条路由 + 18 套提示词 + 契约 + 改虚构模型 + 修迁移）。
**不选 D**：项目并非"基本不可用的空壳"，续写/编辑等旧路径与工程治理是真实且可运行的。

### 8.3 达到"可给作者试用"的最小修复清单（优先级序）
1. **修 BUG-02/01**：把默认模型改成真实模型（`deepseek-chat`/`deepseek-reasoner`），为 18 个新节点补 `model_routes` 播种。
2. **补 18 套节点提示词**（`bootstrap.plan_idea` … `bootstrap.final_humanize`）+ 对应 `OUTPUT_CONTRACTS` / `BOOTSTRAP_OUTPUT_MODELS`（修 BUG-05）。
3. **修 BUG-03**：`nc_commerce_plans` 里 `VARCHAR(36)` 改 `uuid`，与全库一致。
4. **修 BUG-04**：合并 Alembic 多 head。
5. **加一条"真实 provider 端到端冒烟测试"**（非 mock）：从 `plan_idea` 跑到 `write_chapter_draft` 至少产出 1 章真实文本，纳入 CI。
6. 前端：首用引导 + 模型/Key 校验，避免静默走虚构默认。

完成 1–5 后可复审升级为 B（基本可用需小修）。

## 9. 直接回答用户的关键问题

- **现在能给作者用吗？** —— 不能。从零创建新书的主流程会在第 1 个 AI 节点失败（虚构模型）或严重退化（通用兜底提示）。
- **能完整写完一本小说吗？** —— 不能通过主打的"一键生成"闭环完成。仅"在已有书上逐章续写"这条支路在配好真实 key 时可跑。
- **哪些是假实现？** —— 四阶段20节点一键生成（0/18 接线）、商业化套餐订阅（迁移崩溃）、以及一批文档自列"待验收"的采集/发布能力。
- **哪些缺失？** —— 新节点的路由、提示词、输出契约；真实 provider 的端到端回归；开箱即用的部署。
- **距商业产品差多远？** —— 差一个"把已建好的引擎接上真实模型与提示词、修掉阻断级部署 bug、补真实端到端验证"的关键收尾阶段。地基良好，管道未通。
