## 一、项目全局架构分析

**技术栈**：React 19 + TypeScript + Vite（前端，29 组件单页多 Tab）；FastAPI + 裸 psycopg2 连接池（无 ORM）+ PostgreSQL/pgvector + Celery + Redis（后端，130 条路由）；DeepSeek 主 + Claude/OpenAI/Gemini 降级链；Docker Compose（8 服务：pg/redis/api/worker/flower/frontend/migrate/backup）+ Nginx；GitHub Actions（backend/frontend/e2e 三 job）。

**入口与分层**：`app/main.py`（1230 行，含 60+ 内联路由）+ 11 个 api/v1 router；`gateway.complete()`/`complete_stream()` 为 AI 唯一入口（预算断言→路由→严格 Pydantic 输出 Schema→ai_calls 账本→熔断器→降级链→mock 双门禁）；`workers/tasks.py`（1053 行）承载 bootstrap 8 节点线性链、连续章节（分布式锁+generation_key 幂等+审后返工≤2 轮）、批次断点续跑、4 条 beat 定时任务。

**关键调用链**（逐行核实）：
- 扫榜成书：RankingCenter → `/ranking/sources/{s}/scan|import`（Pydantic 严格校验+置信度门禁 0.85）→ snapshots/items → `/analyze`（防注入声明+原创性校验：候选与源榜单标题零重合断言）→ topic_candidates → `/generate-book`（幂等键）→ bootstrap n3~n8 → contents。
- 连续章节：7 层上下文装配 → `complete(client_mutation_id=generation_key)` 可重放 → 落库（partial unique index 幂等）→ 实体/伏笔/时间线/弧线富化（失败记录不阻断）→ 连续性风险报告 → 强制审核+返工≤2 → 摘要。
- 流式编辑：`/contents/{id}/ai/{op}/stream` → `complete_stream`（预算/重放/熔断同语义）→ SSE 增量帧 → 完成写 ai_calls+版本；前端 apiStream 有 401 刷新与降级到非流式。
- 发布：`/publish` → 敏感词前置 → **仅插入 pending 记录，无任何 worker 消费**（见 N-8）。

**高耦合/风险模块**：`workers/tasks.py` 超大文件；无 ORM 双轨（schema 快照契约测试已作为补偿，但本轮仍实锤一处列漂移 N-1）；`main.py` 与 `api/v1/config.py` 存在重复路由且鉴权强度不一致（N-2）。

**死代码/无效代码（本轮新发现）**：`run_events` SSE 端点（引用不存在列，运行必崩，前端实际用 2s 轮询）；`db.row_to_dict` 双重 return；`new_id(prefix)` 忽略 prefix 参数；`_load_prompt_and_route` 中 `task_type.replace("editor_", "editor_")` 无操作；`publish_retry_handler` 以空凭据重试（必然失败）；`_persist_output` 中 `task_type in ("review_7dim","gen_chapter1")` 的 gen_chapter1 分支不可达（外层已限定 review_7dim）。
