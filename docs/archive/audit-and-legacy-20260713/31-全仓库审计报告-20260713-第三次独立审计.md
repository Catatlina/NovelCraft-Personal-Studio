## 五、疑似假实现与占位清单（本轮）

《28》10 项（F-1~F-10）复核：**F-1（settings 假生效）口径已修正**、F-2（SSE 坏死）已修、F-3/F-4（发布占位/空凭据重试）已修、F-5（失败不入账）已修（但引入回归）、F-6（+60ms）已修、F-7（恒定单价）已修、F-8（assembler 预算）已修、F-9（伏笔误报）已修（N-11）、F-10（状态机短 id hack）已修（N-7）。**《28》所列假实现基本清零，整改真实。**

### P0-NEW（本轮新发现，整改引入的回归）— 失败账本毒化幂等重放

| 字段 | 内容 |
|---|---|
| 严重度 | **P0（破坏核心恢复机制）** |
| 位置 | `app/gateway.py` `complete()`/`_record_failed_call()`（新增包装层）+ `_complete_impl()` 重放守卫（`gateway.py:155-162` 仅匹配 `status='succeeded'`） |
| 数据约束 | `ai_calls_client_mutation_uq` UNIQUE `(project_id, client_mutation_id) WHERE client_mutation_id IS NOT NULL` |
| 复现（已实测） | 同一 `client_mutation_id`：①第一次 provider 失败 → `_record_failed_call` 写入 `status='failed'` 行；②provider 恢复后重试 → 重放守卫只查 succeeded（未命中）→ 调 provider 成功 → INSERT succeeded 行 → **`UniqueViolation: duplicate key (project_id, client_mutation_id)`** |
| 实测输出 | `attempt1 failed as expected: ProviderError` / `ledger after failure: [{'status': 'failed'}]` / `attempt2 FAILED — UniqueViolation duplicate key … already exists` |
| 影响范围 | 所有用固定 mutation id 的恢复路径：bootstrap 节点 `bootstrap:{run}:{node}`（tasks.py:111）、章节 review `{gk}:review:{n}:v1`（:642）、rewrite（:677）、榜单分析 `ranking-analysis:{snapshot}`（ranking.py）。即**PRD/M1 验收④「断点续跑」、批次 pending_provider→resume 全部受影响** |
| 触发条件 | 真实 provider 抖动/超时/预算恢复后重试（生产常态）。mock 测试首跑即成功、永不产生失败行，故 437 测试全绿也测不到 |
| 连带缺陷 | `_complete_impl` INSERT 抛异常时未 `conn.close()`，连接以 aborted 事务态泄漏回池（maxconn=20，反复失败可耗尽） |
| 修复方案 | 三选一：①重放守卫改为匹配 `status IN ('succeeded')` 前先 `DELETE`/更新失败行；②失败账本用不同键（如 mutation_id 追加 `:failed:{ts}` 或单独失败表）；③succeeded INSERT 改 `ON CONFLICT (project_id,client_mutation_id) DO UPDATE`。并在失败路径加 `try/finally close`。必须补真实-失败-再成功的回归测试 |

## 六、代码质量 / AI 模块（本轮要点）

- **Q-1（可靠性，真实证据）**：`ranking_market_analysis`/`review_*` 用严格 Pydantic（extra=forbid + min_length + 原创去重）校验真实 LLM 输出，DeepSeek 偶发多字段/少字段/候选撞源 → `502 MARKET_ANALYSIS_OUTPUT_INVALID`。实测同一 T3 用例首跑 fail、复跑 pass。生产表现为间歇性 502。建议：失败自动重试 ≤2（带"上次因 X 失败"反馈）或放宽非关键字段。
- **Q-2**：`_record_failed_call` 的 `except Exception: pass`（吞掉账本写入异常）本意保原始错误，但掩盖了 P0-NEW 的二次 UniqueViolation，排障困难。
- **Q-3**：`run_events` SSE 仍 `while True` 无最大时长/心跳上限（长连接资源）。
- AI 能力实证：真实降级链、严格 Schema、失败账本（除回归外语义正确）、SSRF 白名单、流式、语义嵌入、六维检查、Agent 执行——**工程完整度达"准商业级"**；未达商业级仍差：预算仅 bootstrap 单 scope（无任务/日级）、Schema 脆性致 flaky、Prompt 无模型兼容矩阵、golden case CI 只验定义完整性（`provider_outputs_verified=0`，诚实标注）。
