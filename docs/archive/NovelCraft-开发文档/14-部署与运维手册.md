## 1. 部署架构总览

### 1.1 拓扑图

```
                          ┌─────────────────────────────┐
         Internet         │         Nginx (HTTPS)        │
    ───────────────────►  │  TLS 终止 · 反代 /api        │
                          │  SSE 缓冲关闭 · gzip · 静态  │
                          └───────────────┬─────────────┘
                                          │
                  ┌───────────────────────┼───────────────────────┐
                  ▼                       ▼                       ▼
          ┌──────────────┐       ┌──────────────┐       ┌──────────────────┐
          │   api        │       │  worker-*    │       │  beat            │
          │  FastAPI     │◄─────►│ Celery ×5    │◄─────►│  定时调度        │
          │  uvicorn×2   │       │ plan/generate│       │ (schedules 驱动) │
          └──────┬───────┘       │ /review/    │       └──────────────────┘
                 │               │ publish/    │
                 │               │ system      │
                 │               └──────┬───────┘
                 ▼                      ▼
        ┌────────────────┐      ┌────────────────────┐
        │  PostgreSQL    │      │   Redis            │
        │  + pgvector    │◄────►│ broker·限流·预算   │
        │  唯一事实源    │      │ ·缓存·SSE 发布    │
        └────────────────┘      └────────────────────┘

  数据流向：
  - 用户请求 → Nginx → api → (Postgres / Redis)
  - 长任务   → api 入队 Redis → worker 消费 → 写 Postgres + 发布 SSE 到 Redis → api 推前端
  - beat     → 读 schedules → 派发 system/plan 等定时任务到对应队列
  - 备份     → beat 02:00 触发 bash 任务 → pg_dump | age 加密 → rclone 异地
```

### 1.2 服务角色表

| 服务 | 镜像 | 进程 | 职责 | 数量 |
|---|---|---|---|---|
| nginx | nginx:1.27-alpine | - | TLS 终止、反代、SSE、静态、gzip | 1 |
| api | novelcraft/api | uvicorn，2 worker | REST/SSE、JWT、编排入口 | 1 |
| worker-plan | novelcraft/worker | celery -Q plan | 规划类任务（Producer 产出 WorkflowPlan） | 1 |
| worker-generate | novelcraft/worker | celery -Q generate | 生成/改写/仿写（最重） | 1+ |
| worker-review | novelcraft/worker | celery -Q review | 审核（异构 provider 交叉审） | 1 |
| worker-publish | novelcraft/worker | celery -Q publish | 发布 adapter 调用 | 1 |
| worker-system | novelcraft/worker | celery -Q system | 备份、清理、知识入库、热点采集 | 1 |
| beat | novelcraft/worker | celery beat | 定时任务派发 | 1 |
| postgres | pgvector/pgvector:pg16 | postgres | 唯一事实源 + 向量 | 1 |
| redis | redis:7-alpine | redis-server | broker/限流/预算/缓存/SSE | 1 |

### 1.3 网络与卷

- 网络：`nc_net`（bridge 单网），所有服务同网，仅 nginx 暴露 80/443 给宿主机。
- 卷：
  - `pg_data` → postgres `/var/lib/postgresql/data`
  - `redis_data` → redis `/data`
  - `nginx_ssl` → nginx `/etc/nginx/ssl`（证书）
  - `uploads` → api `/app/uploads`（导入文件暂存，可选）
  - `backups` → 宿主机 `/opt/novelcraft/backups`（本地留 7 天 + rclone 异地 30 天）

## 2. docker-compose.yml 完整示例

生产用 `docker-compose.yml`（与 dev 区分）。占位变量全部来自 `.env`。

```yaml
# docker-compose.yml —— NovelCraft Personal Studio 生产栈
# 资源基线：4C8G。generate worker 并发 = min(各 provider RPM 限额)
# 用法：docker compose --env-file .env up -d

services:

  nginx:
    image: nginx:1.27-alpine
    restart: unless-stopped
    depends_on:
      api:
        condition: service_healthy
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/conf.d:/etc/nginx/conf.d:ro
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - nginx_ssl:/etc/nginx/ssl:ro
      - uploads:/var/www/uploads:ro
    networks: [nc_net]
    healthcheck:
      test: ["CMD", "wget", "-q", "-O", "-", "http://127.0.0.1/healthz-nginx"]
      interval: 30s
      timeout: 5s
      retries: 3
    deploy:
      resources:
        limits: { cpus: "1.0", memory: 256M }

  api:
    image: ${REGISTRY:-ghcr.io/your-org}/novelcraft/api:${TAG:-latest}
    restart: unless-stopped
    env_file: .env
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    volumes:
      - uploads:/app/uploads
    networks: [nc_net]
    expose:
      - "8000"
    healthcheck:
      test: ["CMD", "python", "-c", "import urllib.request,sys; sys.exit(0 if urllib.request.urlopen('http://127.0.0.1:8000/healthz').status==200 else 1)"]
      interval: 20s
      timeout: 5s
      retries: 5
      start_period: 30s
    deploy:
      resources:
        limits: { cpus: "2.0", memory: 2g }

  worker-plan: &worker
    image: ${REGISTRY:-ghcr.io/your-org}/novelcraft/worker:${TAG:-latest}
    restart: unless-stopped
    env_file: .env
    command: celery -A workers.celery_app worker -Q plan -c 2 --max-tasks-per-child 200
    depends_on:
      postgres: { condition: service_healthy }
      redis: { condition: service_healthy }
    networks: [nc_net]
    healthcheck:
      test: ["CMD", "celery", "-A", "workers.celery_app", "inspect", "ping", "-q", "plan"]
      interval: 30s
      timeout: 10s
      retries: 3
    deploy:
      resources:
        limits: { cpus: "1.0", memory: 1g }

  worker-generate:
    <<: *worker
    command: celery -A workers.celery_app worker -Q generate -c ${GENERATE_CONCURRENCY:-2} --max-tasks-per-child 100
    deploy:
      resources:
        limits: { cpus: "2.0", memory: 2g }

  worker-review:
    <<: *worker
    command: celery -A workers.celery_app worker -Q review -c 2 --max-tasks-per-child 200
    deploy:
      resources:
        limits: { cpus: "1.0", memory: 1g }

  worker-publish:
    <<: *worker
    command: celery -A workers.celery_app worker -Q publish -c 2 --max-tasks-per-child 200
    deploy:
      resources:
        limits: { cpus: "0.5", memory: 768M }

  worker-system:
    <<: *worker
    command: celery -A workers.celery_app worker -Q system -c 2 --max-tasks-per-child 200
    deploy:
      resources:
        limits: { cpus: "0.5", memory: 768M }

  beat:
    image: ${REGISTRY:-ghcr.io/your-org}/novelcraft/worker:${TAG:-latest}
    restart: unless-stopped
    env_file: .env
    command: celery -A workers.celery_app beat -S redbeat.RedBeatScheduler
    depends_on:
      redis: { condition: service_healthy }
    networks: [nc_net]
    healthcheck:
      test: ["CMD", "celery", "-A", "workers.celery_app", "inspect", "ping", "-q", "celery"]
      interval: 30s
      timeout: 10s
      retries: 3
    deploy:
      resources:
        limits: { cpus: "0.25", memory: 256M }

  postgres:
    image: pgvector/pgvector:pg16
    restart: unless-stopped
    env_file: .env
    environment:
      POSTGRES_DB: ${DB_NAME}
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes:
      - pg_data:/var/lib/postgresql/data
      - ./postgres/init:/docker-entrypoint-initdb.d:ro
    networks: [nc_net]
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER} -d ${DB_NAME}"]
      interval: 10s
      timeout: 5s
      retries: 5
    deploy:
      resources:
        limits: { cpus: "2.0", memory: 2g }

  redis:
    image: redis:7-alpine
    restart: unless-stopped
    command: ["redis-server", "--appendonly", "yes", "--maxmemory", "1g", "--maxmemory-policy", "allkeys-lru"]
    volumes:
      - redis_data:/data
    networks: [nc_net]
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 5
    deploy:
      resources:
        limits: { cpus: "0.5", memory: 1g }

volumes:
  pg_data:
  redis_data:
  nginx_ssl:
  uploads:

networks:
  nc_net:
    driver: bridge
```

> 说明：`beat` 使用 `redbeat`（Redis 后端），避免重复 beat 实例抢任务；生产环境只跑一个 beat。
> 资源限制为 4C8G 基线下的保守分配，可按 VPS 规格等比例放大（见 §9）。

## 4. Nginx 配置要点

### 4.1 主配置（nginx.conf，节选）

```nginx
user  nginx;
worker_processes  auto;
events { worker_connections 1024; }

http {
    include       /etc/nginx/mime.types;
    default_type  application/octet-stream;

    # gzip
    gzip on;
    gzip_comp_level 6;
    gzip_min_length 1024;
    gzip_types text/plain text/css application/json application/javascript application/xml image/svg+xml;

    # 上游
    upstream api_upstream {
        server api:8000 max_fails=3 fail_timeout=15s;
    }

    # 大 body（导入 PDF/DOCX 知识）
    client_max_body_size 50m;

    include /etc/nginx/conf.d/*.conf;
}
```

### 4.2 站点配置（conf.d/studio.conf）

```nginx
server {
    listen 80;
    server_name studio.example.com;
    # 强制跳 HTTPS（ACME 验证可临时放开）
    location /.well-known/acme-challenge/ { root /var/www/certbot; }
    location / { return 301 https://$host$request_uri; }
}

server {
    listen 443 ssl;
    http2 on;
    server_name studio.example.com;

    ssl_certificate     /etc/nginx/ssl/fullchain.pem;
    ssl_certificate_key /etc/nginx/ssl/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;

    # 静态前端（PWA）
    location / {
        root /var/www/studio;
        try_files $uri $uri/ /index.html;
        add_header Cache-Control "public, max-age=3600";
    }
    location /assets/ {
        root /var/www/studio;
        add_header Cache-Control "public, max-age=31536000, immutable";
    }

    # API 反代
    location /api/ {
        proxy_pass http://api_upstream/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 300s;
    }

    # SSE：必须关闭缓冲，否则进度推不下来
    location /api/v1/events {
        proxy_pass http://api_upstream/v1/events;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_buffering off;          # 关键
        proxy_cache off;              # 关键
        chunked_transfer_encoding on;
        proxy_read_timeout 3600s;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 健康检查（nginx 侧）
    location /healthz-nginx { return 200 "ok"; }
}
```

**要点回顾：**
- TLS 终止在 Nginx，后端全程 HTTP（同网可信）。
- SSE 路径 `proxy_buffering off` + `proxy_cache off` + `Connection ""`，否则事件流被缓冲。
- 静态资源走 `stale-while-revalidate` 由前端 Service Worker 控制，Nginx 仅给长缓存。
- 上线前用 `nginx -t` 校验配置。

## 5. 首次部署步骤

1. **准备 VPS**：Ubuntu 22.04+，4C8G，开放 22/80/443，域名解析到该 IP。
2. **装 Docker + Compose**：
   ```bash
   curl -fsSL https://get.docker.com | sh
   docker compose version   # 需 v2.20+
   ```
3. **拉代码**：
   ```bash
   git clone git@github.com:your-org/novelcraft.git /opt/novelcraft
   cd /opt/novelcraft && git checkout <release-tag>
   ```
4. **生成密钥**：
   ```bash
   python -c "import secrets,base64; print(base64.b64encode(secrets.token_bytes(32)).decode())"  # JWT
   python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"        # FERNET
   age-keygen -o /opt/novelcraft/age.key   # 备份此私钥到密码管理器
   ```
5. **写 `.env`**：复制 `.env.example` → `.env`，填入上述所有变量，权限 `chmod 600 .env`。
6. **TLS 证书**：用 `certbot` 或 `acme.sh` 申请，放到 `./nginx/ssl/`（fullchain.pem / privkey.pem）。
7. **起数据库与 Redis**（先仅这两项）：
   ```bash
   docker compose up -d postgres redis
   ```
8. **初始化数据库（迁移 + seed）**：
   ```bash
   docker compose run --rm api alembic upgrade head
   docker compose run --rm api python -m app.seed   # 默认项目/预设工作流/内置 Prompt/敏感词
   ```
9. **起全栈**：
   ```bash
   docker compose up -d
   docker compose ps        # 全部 healthy
   curl -f https://studio.example.com/api/v1/healthz
   ```
10. **验证**：浏览器打开 `https://studio.example.com`，注册首个账号（owner），跑一条预设工作流（如「黄金三章」试跑样例）。

## 6. 升级与回滚流程（make deploy）

`Makefile` 封装一键升级，语义：**备份 → pull → migrate → up → healthcheck → 失败回滚上一 tag**。

```makefile
# Makefile（节选）
TAG ?= latest
PREV_TAG := $(shell git describe --tags --abbrev=0)

.PHONY: deploy
deploy: backup pull migrate up healthcheck

backup:
	@bash scripts/backup.sh --tag $(TAG)        # 升级前强制备份

pull:
	docker compose pull

migrate:
	docker compose run --rm api alembic upgrade head

up:
	docker compose up -d

healthcheck:
	@bash scripts/wait_health.sh --timeout 120
	@curl -f https://studio.example.com/api/v1/healthz || $(MAKE) rollback

rollback:
	@echo "升级健康检查失败，回滚到 $(PREV_TAG)"
	@TAG=$(PREV_TAG) docker compose up -d
	@docker compose run --rm api alembic downgrade -1 || true
	@echo "已回滚，请排查原因"
```

### 6.1 升级标准动作

```bash
# 在发布分支打 tag 后，手动触发：
git checkout v1.2.3
make deploy TAG=v1.2.3
```

### 6.2 升级前后检查清单

- [ ] 升级前自动备份完成（且异地 rclone 同步成功）
- [ ] `alembic upgrade head` 无报错（迁移脚本可降级）
- [ ] 全部服务 `healthy`
- [ ] `/healthz` 返回 200，队列深度为 0 或回落
- [ ] 跑一条 smoke 工作流（golden case 通过）
- [ ] Telegram 收到「部署成功」提示

### 6.3 回滚

- 自动回滚：healthcheck 失败触发 `make rollback`（拉回上一 tag 镜像 + 迁移 downgrade -1）。
- 手动回滚：`TAG=<prev> docker compose up -d`；若数据已迁移且不可降，则走备份恢复（§7）。

## 7. 备份与恢复（RPO≤24h / RTO≤1h）

### 7.1 备份机制

- 触发：beat 每日 `02:00` 派发 `system` 队列任务 `backup_db`。
- 流程：`pg_dump --clean --no-owner` → `age` 加密 → 本地 `backups/` 留 7 天 → `rclone copy` 异地（对象存储/S3 兼容）→ 保留 30 天。
- 加密：所有 dump 用 `AGE_PUBLIC_KEY` 加密，私钥只存运维密码管理器，**不进镜像、不进 `.env`**。

```bash
# scripts/backup.sh（核心片段）
set -euo pipefail
TS=$(date +%Y%m%d_%H%M%S)
OUT="/backups/nc_${TS}.sql.gz.age"
pg_dump "$DATABASE_URL" | gzip | age -R "$AGE_PUBLIC_KEY" > "$OUT"
rclone copy "$OUT" "$RCLONE_REMOTE/"
# 本地只留 7 天
find /backups -name 'nc_*.sql.gz.age' -mtime +7 -delete
# 异地 30 天由 rclone remote 生命周期策略负责
echo "backup done: $OUT"
```

### 7.2 恢复演练（每月入日历）

```bash
# 演练：在隔离环境恢复，校验可读取
LATEST=$(rclone lsf "$RCLONE_REMOTE/" --files-only | sort | tail -1)
rclone copy "$RCLONE_REMOTE/$LATEST" /tmp/restored.age
age --decrypt -i /path/to/age.key /tmp/restored.age | gunzip | \
  psql "$RESTORE_DB_URL"   # 恢复到临时库校验
# 校验：表数量、最新一条 ai_calls、用户可登录
```

### 7.3 真实恢复步骤（RTO 目标 ≤1h）

1. 停止写服务避免数据漂移：`docker compose stop api worker-generate worker-review worker-publish worker-plan`。
2. 拉取异地最新备份：`rclone copy "$RCLONE_REMOTE/$LATEST" /tmp/restore.age`。
3. 解密 + 导入：`age -d -i age.key /tmp/restore.age | gunzip | psql "$DATABASE_URL"`。
4. 重建搜索索引/向量（若有外部索引）：`docker compose run --rm api python -m app.reindex`。
5. 起服务：`docker compose up -d` → healthcheck。
6. 通知：Telegram 报「恢复完成，时间点=<备份时间>」。

> 指标：RPO ≤ 24h（每日 02:00 全量）；RTO ≤ 1h（恢复演练实测达标即为有效）。

### 7.4 备份失败告警

beat 任务失败 / rclone 退出非 0 → 触发 Telegram 告警「备份失败：<原因>」，**备份失败等同线上事故**处理。

## 8. 监控与告警

### 8.1 健康检查 `/healthz`

聚合检查项，返回整体状态 + 分项：

```json
{
  "status": "ok",
  "db": "ok",
  "redis": "ok",
  "queue_depths": {"plan": 0, "generate": 1, "review": 0, "publish": 0, "system": 0},
  "disk_usage_pct": 41
}
```

- DB：`SELECT 1` 成功。
- Redis：`PING` 成功。
- 队列深度：各队列 `LLEN`，`>= 50` 标黄，`>= 200` 标红。
- 磁盘：`shutil.disk_usage` 根分区，超 `ALERT_ON_DISK_WATERMARK` 告警。

### 8.2 日志（structlog JSON）

所有服务统一 JSON 日志，本地滚动（`logging.handlers.RotatingFileHandler`，单文件 50MB × 5）。不上 ELK（1~5 人规模）。关键字段：`ts / level / logger / event / run_node_id / task_type / provider / cost`。

```python
import structlog
log = structlog.get_logger()
log.info("ai_call_done", run_node_id=rn, provider="deepseek", cost=0.0021, latency_ms=812)
```

### 8.3 Telegram 告警触发条件

| 触发 | 条件 | 级别 |
|---|---|---|
| 任务失败 | Celery task 进入 FAILED 且非预期重试 | ⚠️ |
| 预算触顶 | 单日预算计数达上限，新任务被拒 | ⚠️ |
| Provider 全降级 | 某 task_type 主+备全失败，run_node 置 PENDING_PROVIDER | 🔴 |
| 磁盘水位 | `disk_usage_pct >= 85` | 🔴 |
| 备份失败 | beat 备份任务退出非 0 或 rclone 失败 | 🔴 |
| 队列积压 | 任一队列 `LLEN >= 200` 持续 5 分钟 | ⚠️ |
| 部署成功 | `make deploy` healthcheck 通过 | ℹ️ |

告警格式示例：

```
🔴 [PROVIDER_DEGRADED] task_type=chapter_generate 全部 provider 降级
run_node=abc123 已置 PENDING_PROVIDER，恢复后自动重试。
```

## 9. 五队列 Worker 并发与扩容

### 9.1 队列划分与并发基线

| 队列 | 任务类型 | 并发基线 | 说明 |
|---|---|---|---|
| plan | Producer 规划、WorkflowPlan 生成 | 2 | 轻量 |
| generate | 草稿/改写/仿写/润色（最重） | `GENERATE_CONCURRENCY` | 见下 |
| review | 审核（异构 provider 交叉审） | 2 | 与 generate 异构 |
| publish | 发布 adapter 调用 | 2 | IO 密集 |
| system | 备份/清理/知识入库/热点采集 | 2 | 后台 |

**generate worker 并发黄金法则：**

```
GENERATE_CONCURRENCY = min(各 provider RPM 限额)
```

例：DeepSeek RPM=20、Claude RPM=10、OpenAI RPM=10、Gemini RPM=10 → 并发上限取 10。
再叠加令牌桶限流（Redis），保证不超 provider 配额，避免 429 雪崩。

### 9.2 扩容策略（垂直优先）

1. **垂直优先**：先在单 VPS 内加 CPU/内存，提高各 worker 的 `cgroup` 上限与 `-c` 并发。
2. 若 generate 仍是瓶颈：单独给 `worker-generate` 升配（更大内存，因为它是 LLM 输出解析 + 上下文装配最重）。
3. **水平仅限 generate**：当单 VPS 已到顶且预算允许，可把 `worker-generate` 拆到第二台 VPS，只连同一 Redis/Postgres（同网或专线）。其余服务仍单节点。
4. 不引入 K8s / 服务网格（架构红线，见 ADR-007）。

## 10. 常见故障排查手册

| 现象 | 可能原因 | 排查 | 处置 |
|---|---|---|---|
| `/healthz` 返回 503 | DB 或 Redis 不可达 | `docker compose logs postgres`；`redis-cli ping` | 重启对应服务；检查密码/网络 |
| 前端进度条不动（SSE 无事件） | Nginx SSE 缓冲未关 | `nginx -T` 看 `proxy_buffering` | 改配置 `proxy_buffering off`，reload |
| 生成任务卡 PENDING_PROVIDER | 所有 provider 降级 | `/healthz` + 日志 grep `degraded` | 检查 provider key/余额；恢复后自动重试 |
| 429 限流雪崩 | 并发超过 provider RPM | `redis-cli` 看令牌桶 | 降低 `GENERATE_CONCURRENCY`；调 model_routes |
| 队列越积越多 | worker 崩溃/未起 | `docker compose ps`；`celery inspect active` | 重启 worker；看 traceback |
| 迁移失败 `alembic` 报错 | 迁移与 schema 冲突 | `docker compose run api alembic heads` | 修复迁移或回滚上一 tag |
| 磁盘 100% | 备份/日志/向量膨胀 | `df -h`；`du -sh /var/lib/postgresql` | 清理旧备份；扩盘；调保留天数 |
| JWT 登录即失效 | JWT_SECRET 不一致 | 比对各容器 env | 统一 `.env`，重启 api |
| 平台账号发布报鉴权失败 | Fernet 解密失败/密钥变更 | `docker compose logs worker-publish` | 确认 `FERNET_KEY` 未变；重填账号 |
| 备份未上传 | rclone 配置/网络 | `rclone lsd remote:` | 修 rclone.conf；重跑 `backup.sh` |
| 相似度检测误杀正常仿写 | 阈值过严 | 查 `ai_calls.meta.sim` | 调 `config` 阈值（需人工确认变更） |
| 自动发布卡在待点击 | 国内全自动未开启（符合预期） | 设置页确认开关 | 按设计即半自动，属正常 |

## 11. 灾备与应急预案

### 11.1 故障分级

| 级 | 定义 | 响应 |
|---|---|---|
| P0 | 全站不可用 / 数据丢失风险 | 立即恢复备份，通知全员 |
| P1 | 核心能力不可用（生成/发布挂） | 限时 1h 修复或回滚 |
| P2 | 单队列积压 / 单 provider 降级 | 监控观察，自动重试 |
| P3 | 非核心告警 | 排期处理 |

### 11.2 灾难场景与处置

- **Postgres 数据损坏**：停止写入 → 异地备份恢复（§7.3）→ 重建向量索引 → 起服务。
- **VPS 整机故障**：新开 VPS → 装 Docker → 拉代码与 `.env`（从密码管理器恢复）→ 恢复备份 → DNS 切 IP。
- **Provider 全部封禁**：切 model_routes 到可用 provider（Redis 热更新，无需发版）→ 告警提示。
- **密钥泄露**：立即在 provider 后台吊销 → 改 `.env` 全部密钥 → 滚动重启 → 审计 `ai_calls`（排查异常消耗）。

### 11.3 应急联系人 / 演练

- 每季度一次「备份恢复演练」（§7.2）+ 一次「VPS 故障切换演练」，结论归档 `docs/reports/`。
- 运维负责人 A/B 角，密钥分片存密码管理器。
