# 星禾AI工作台 · AI代码生成门禁

> 版本：V1.0 | 日期：2026-07-20 | 角色：质量负责人
>
> ⚠️ 所有AI生成的代码必须经过此门禁。

---

## 一、AI代码生成流程

```
分析需求
    ↓
设计方案（输出书面方案）
    ↓
生成代码
    ↓
──── 门禁检查 ────
    ↓
安全检查（Secret/注入/依赖）
    ↓
测试验证（单元/集成/E2E）
    ↓
人工Review
    ↓
合并
```

---

## 二、门禁检查清单

### 2.1 功能完整性

| 检查项 | 标准 |
|--------|------|
| □ 功能是否完整可用 | 不是Demo、不是Mock |
| □ 数据链路是否完整 | UI → API → Service → DB → Result |
| □ 异常处理是否完备 | try/catch、错误提示、降级方案 |
| □ 空/加载/错误状态 | 三态齐全 |

### 2.2 代码质量

| 检查项 | 标准 |
|--------|------|
| □ TypeScript类型完整 | `tsc --noEmit` 0错误 |
| □ ESLint通过 | 0 Error |
| □ Prettier格式化 | 已格式化 |
| □ 无硬编码颜色/数值 | 使用Design Token |
| □ 重复代码已抽离 | 出现2次已组件化 |

### 2.3 安全

| 检查项 | 标准 |
|--------|------|
| □ 无硬编码密钥 | gitleaks通过 |
| □ 无SQL注入 | 参数化查询 |
| □ 无XSS风险 | 输出转义 |
| □ 输入校验 | 所有外部输入校验 |
| □ 权限校验 | authz.py鉴权 |

### 2.4 兼容性

| 检查项 | 标准 |
|--------|------|
| □ 未删除现有API | 只做加法 |
| □ 未破坏现有数据 | Migration可逆 |
| □ 未改变现有行为 | 回归测试通过 |

---

## 三、AI代码红线（Block级别）

以下情况**直接Block，不允许合并**：

| ❌ 红线 | 说明 |
|--------|------|
| 删除已有功能 | 任何功能、接口、数据 |
| 用假数据替代真实逻辑 | Mock/硬编码替代真实API |
| 绕过AI Engine直接调Provider | 必须通过统一Engine |
| 硬编码密钥/密码 | gitleaks自动拦截 |
| 破坏API契约 | 改变现有请求/响应格式 |
| 跳过测试 | 新功能必须有测试 |
| 静默吞错误 | try/catch不能空 |

---

## 四、AI代码黄线（Warning级别）

| ⚠️ 黄线 | 说明 |
|--------|------|
| 未更新文档 | 功能变更后必须更新对应文档 |
| 重复代码 | 复用已有组件/Services |
| 硬编码样式 | 使用Design Token |
| 未处理边界情况 | 空值/极限值/并发 |
| 性能问题 | N+1查询、未建索引 |
| 日志不足 | 关键操作无日志 |

---

## 五、假功能检测

AI生成的代码必须通过假功能检测：

| 检测项 | 方法 |
|--------|------|
| 按钮是否有真实onClick | 代码审查 |
| API是否真实存在 | 对照API_SPEC.md |
| 数据是否写入数据库 | 对照DATABASE_SCHEMA.md |
| 任务是否真实创建 | Celery任务队列验证 |
| 结果是否真实返回 | E2E测试验证 |

**全链路验证：UI → API → Service → Database → Task → Result**

---

## 六、验收标准

AI生成代码合并前必须满足：

```
✅ 功能完整可用（非Demo/非Mock）
✅ 测试通过（单元 + 集成 + E2E核心链路）
✅ 安全检查通过（Secret/注入/依赖）
✅ 兼容性检查通过（未破坏现有功能）
✅ 文档已更新（涉及的功能模块文档）
✅ 代码符合规范（ESLint/Prettier/TypeScript/ruff）
```
